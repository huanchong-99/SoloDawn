# AREA R4 — Quality Gate Engine + Audit/Scoring System

Research for the feature: **after solution confirmation, a secondary popup that DISPLAYS the built-in quality-gate rules and lets users edit / add / delete them (custom per-project rules), defaulting to built-ins.**

Verified against source at commit `9995e65eb` (branch `main`). All file:line refs verified with Read/Grep.

---

## 0. TL;DR — what a "rule" is and the cleanest place to make it editable

There are **two completely separate "rule" systems** in this repo. The feature description ("display built-in quality-gate rules, edit/add/delete") maps to **System A (the YAML gate conditions)**, but the audit popup wired at confirm-time today is **System B (the scoring rubric)**. They are NOT the same model and NOT currently editable in the UI.

| | System A: Quality Gate (the three gates) | System B: Audit Scoring Plan |
|---|---|---|
| What a "rule" is | A `Condition` = `metric` + `operator (GT/LT)` + `threshold` | An `AuditDimensionSpec` = name + max_score + `criteria: Vec<String>` |
| Built-ins live in | `quality/quality-gate.yaml` (compile-time embedded via `build.rs` → `BUNDLED_CENTRAL_POLICY`) | `crates/services/src/services/orchestrator/audit_principles.rs` (`BUILTIN_AUDIT_PRINCIPLES` const + `default_audit_plan()`) |
| Data model | `crates/quality/src/config.rs` (`QualityGateConfig`, `GateDefinition`, `ConditionConfig`) | `crates/services/.../orchestrator/types.rs` (`AuditPlan`, `AuditDimensionSpec`, `AuditMode`) |
| Per-project override TODAY | Yes, but **file-based only** — a repo-local `quality/quality-gate.yaml`. No DB, no API, no UI. | Yes — stored as opaque JSON in `planning_draft.audit_plan` / `workflow.audit_plan`, generated by an LLM at confirm time. No structured edit UI. |
| Reaches the running terminal via | `QualityEngine::from_project(working_dir)` in `agent.rs` (reads repo file) | `workflow.audit_plan` JSON → `build_scoring_review_prompt` in `agent.rs` |
| Serializable | Yes (`serde` + `serde_yaml`) | Yes (`serde_json`) |
| ts-rs generated TS type | **No** (quality crate has no `TS` derives on config types; only `QualityRunSummary/Detail` are ts-rs'd) | **No** (carried as `auditPlan: string` opaque JSON in `api.ts`) |

**Cleanest place to persist per-project editable gate rules (System A):** a new DB table keyed by `project_id` (mirroring how `quality_policy_snapshot` already stores `config_yaml`), surfaced through `QualityGateConfig` so the engine can prefer DB → repo-file → bundled. The struct is already fully `serde`-round-trippable (see `config.rs` test `test_yaml_roundtrip`), so the data model needs **no change** — only a persistence + load-order + API + UI layer. **This backing does NOT exist today (mark: unavailable).**

If the feature instead means the *audit scoring* popup (System B), the editable surface is `AuditPlan.dimensions[].criteria[]` + `pass_threshold`, but today it is LLM-generated free text with only an upload/delete-doc UI (`AuditDocPanel.tsx`), not a structured rule editor.

## 1. System A — Quality Gate Engine (`crates/quality/src/**`)

### 1.1 The rule data model (the gate condition)

The atomic unit the three gates evaluate is a **`Condition`**, NOT the deep static-analysis rules.

`crates/quality/src/gate/condition.rs`:
- `Operator` enum (L15-23): `GreaterThan` (`"GT"`) / `LessThan` (`"LT"`). `from_db_value` / `to_db_value` (L27-41) already serialize to/from DB strings.
- `Condition` struct (L71-82): `metric: MetricKey`, `operator: Operator`, `error_threshold: String`, `use_variation: bool`. **Equality + Hash are by `metric` only** (L129-142) — so a gate can hold at most one condition per metric.

`crates/quality/src/metrics.rs`:
- `MetricKey` enum (L11-169) — **44 fixed variants**, each `#[serde(rename=...)]` to a snake_case key (e.g. `cargo_check_errors`, `clippy_warnings`, `tsc_errors`, `line_coverage`, `secrets_detected`, `builtin_rust_critical`, `test_authenticity_issues`, `quality_gate_empty_scan`). `as_str()` (L173) and `display_name()` (L223) give the machine key and human label. **This is a closed enum — adding a new metric requires Rust code; a user cannot define a brand-new metric from the UI.** This is the single biggest constraint on "add a rule": users can add/remove/retune conditions over the existing 44 metrics, but cannot invent a new metric.

### 1.2 Config / built-in defaults

`crates/quality/src/config.rs`:
- `QualityGateConfig` (L20-36): `mode`, `terminal_gate`, `branch_gate`, `repo_gate` (each a `GateDefinition`), `providers: ProvidersConfig`, `sonar: SonarConfig`. **This is the top-level serializable policy object.**
- `QualityGateMode` (L41-53): `off | shadow | warn | enforce` (`#[serde rename_all="lowercase"]`, default `Shadow`).
- `GateDefinition` (L56-62): `name: String` + `conditions: Vec<ConditionConfig>`.
- `ConditionConfig` (L65-73): YAML-friendly shape — `metric: MetricKey`, `operator: String`, `threshold: String`. `to_condition()` (L77) converts to `Condition`. **This is the exact shape a UI rule-editor would serialize.**
- `ProvidersConfig` (L84-114): 13 bool toggles (rust, frontend, repo, security, sonar, builtin_rust, builtin_frontend, builtin_common, coverage, completeness, delivery_readiness) — also user-relevant for an editor.
- `BUNDLED_CENTRAL_POLICY` (L158-159): `include_str!(concat!(env!("OUT_DIR"), "/quality-gate.yaml"))` — the built-in default, embedded at compile time.
- **Load order** — `load_from_project(project_root)` (L181-217): (1) repo-local `quality/quality-gate.yaml` / `.yml` / `.quality-gate.yaml`; (2) `BUNDLED_CENTRAL_POLICY`; (3) `default_config()` only if bundled fails to parse. **This is the function to extend for DB-backed per-project rules — insert a DB lookup as priority 0.**
- `default_config()` (L220-311): the lenient `Shadow` fallback (smaller condition set).
- `get_gate(level)` (L314-328): builds a `QualityGate` for a level.

`build.rs` (`crates/quality/build.rs`): copies the repo `quality/quality-gate.yaml` into `OUT_DIR` at compile time; falls back to an inline minimal enforce policy if the source can't be found (out-of-tree builds). Resolution: `SOLODAWN_QUALITY_POLICY` env → `../../quality/quality-gate.yaml` → cwd-relative.

`quality/quality-gate.yaml` (the actual built-in content the popup must DISPLAY):
- `mode: enforce` (L12).
- `terminal_gate` (L17-72): 17 conditions (cargo_check_errors, clippy_errors, tsc_errors, rust/frontend test failures, frontend_test_deps_missing, builtin_rust/frontend_critical, secrets_detected, test_file_absence, stub_test_count, coverage_exclusion_issues, redos_risks, test_authenticity_issues, project_convention_issues, runtime_security_smells).
- `branch_gate` (L83-148): 21 conditions (adds clippy_warnings, fmt_violations, test_failures, builtin_*_issues>10, duplicated_blocks>5, rust_cyclomatic_complexity>25, line_coverage<60, todo_density>5).
- `repo_gate` (L153-231): 26 conditions (adds sonar_blocker/critical, security_issues, builtin_common_issues>5, duplicated_blocks>3, line_coverage<80, etc.).
- `providers:` (L234-246) and `sonar:` (L249-253).
- **Design note baked into the YAML (L78-82, L168-171):** `eslint_errors` is deliberately excluded from gates — ESLint severity is advisory/model-dependent (see §1.5).

`quality/profiles/enforce-mode.yaml` and `warn-mode.yaml`: **overlay files with a DIFFERENT shape** (`notify`, `thresholds.max_critical`, `required_checks`). Verified via Grep these are referenced ONLY by `scripts/quality/sync-quality-profile.*`, CI (`.github/workflows/ci-quality.yml`), and docs — **they are NOT parsed by the Rust `QualityGateConfig` loader.** Do not treat them as part of the engine data model.

### 1.3 The gate model + evaluation

`crates/quality/src/gate/mod.rs`:
- `QualityGate` (L25-33): `id`, `name`, `conditions: Vec<Condition>`. `new` / `with_id`.
- `evaluate(results)` (L59-75) → `QualityGateDecision` (status OK/Warn/Error; any Error ⇒ Error).
- `QualityGateDecision` (L79-89): `is_passed()` = only `Ok` passes (Warn does NOT pass in enforce).
- `QualityGateLevel` (L116-124): `Terminal | Branch | Repo`.
- Other gate files: `condition.rs` (above), `evaluator.rs` (`ConditionEvaluator::evaluate_all`), `result.rs` (`EvaluationResult`, `MeasureValue`), `status.rs` (`QualityGateStatus`, `Level`).

### 1.4 The execution engine

`crates/quality/src/engine.rs`:
- `QualityEngine` (L39-42): `config` + `providers`.
- `from_project(project_root)` (L79-141): loads config, instantiates enabled providers. **This is the only constructor the orchestrator uses.**
- `run(...)` (L149-396): concurrently runs providers, aggregates `ProviderReport`s, filters conditions to metrics that an active provider supports (`G16-005`, L256-281), evaluates, and has an **empty-scan fail-closed** path (L296-350) that blocks in enforce mode when targets exist but no provider matched any condition.
- `run_with_scope(...)` (L400-407) + `QualityScope` (L50-70): `IntroducedOnly{merge_base,head,files}` / `TaskBranch` / `RepoFinal`.

### 1.5 The "deep" static-analysis rules (a SECOND, separate layer)

`crates/quality/src/rules/**` (~40 files): per-language lint rules (rust/, typescript/, common/), e.g. `cyclomatic_complexity.rs`, `secret_detection.rs`, `any_usage.rs`. These are **executed inside the builtin providers** (`provider/builtin_rust.rs`, `builtin_frontend.rs`, `builtin_common.rs`) and roll up into the metric counters (`builtin_rust_issues`, etc.). `crates/quality/src/rule.rs` defines `RuleType` (Bug/Vulnerability/CodeSmell/SecurityHotspot, L9-18), `Severity` (Info..Blocker, L46-57), and the model-agnostic `cap_for_advisory` / `AnalyzerSource::severity_origin` machinery (L124-222) — this is the "ESLint can't gate" principle. **These rules are hard-coded Rust; they are NOT user-editable and out of scope for a thresholds editor, but a "providers on/off" toggle (ProvidersConfig) is the realistic editable surface here.**

`crates/quality/src/issue.rs` — `QualityIssue` (L14-46): per-finding record (rule_id, rule_type, severity, source, message, file/line, effort, is_new, context). Maps 1:1 to the `quality_issue` DB table.

## 2. System B — Audit / Scoring System (commit `11ac066d2`)

End-to-end flow: **confirm-plan → generate AuditPlan (LLM) → store on draft → copy to workflow on materialize → orchestrator builds scoring prompt at acceptance review.**

### 2.1 Data model

`crates/services/src/services/orchestrator/types.rs`:
- `AuditPlan` (L582-593): `mode: AuditMode`, `dimensions: Vec<AuditDimensionSpec>`, `pass_threshold: f64`, `generated_at: String`, `raw_principles: String` (**the actual text injected into the review LLM prompt**).
- `AuditMode` (L595-601): `Builtin | Merged | Custom` (`#[serde rename_all="snake_case"]`).
- `AuditDimensionSpec` (L603-610): `name`, `name_zh`, `max_score`, `criteria: Vec<String>`, `sub_dimensions: Option<Vec<...>>` (recursive — code_quality has architecture/standards/security sub-dims). **This is the structured, editable shape a popup could render.**
- `AuditScoreResult` / `AuditDimensions` / `DimensionScore` / `CodeQualityScore` (L548-795): the LLM's scoring *output* (not the plan). `PASS_THRESHOLD = 90.0` (L613). `parse()` extracts JSON, sums dimensions, sets pass/fail. `to_acceptance_result()` bridges to legacy `AcceptanceReviewResult`.

### 2.2 Built-in defaults

`crates/services/src/services/orchestrator/audit_principles.rs`:
- `BUILTIN_AUDIT_PRINCIPLES` (L15-108): the full 100-point rubric text (3 veto rules + 5 dimensions: Buildability 20, Functional Completeness 25, Code Quality 30 [arch/standards/security 10 each], Test Quality 15, Engineering & Docs 10).
- `default_audit_plan()` (L114-122): the structured fallback.
- `builtin_dimension_specs()` (L125-221): the structured 5-dimension default (the data a structured editor would seed from).

### 2.3 Generation pipeline

`crates/services/src/services/orchestrator/audit_plan.rs`:
- `generate_audit_plan(llm, requirement_summary, technical_spec, user_doc, mode)` (L22-70): LLM-tailors the plan; **fail-closed → `default_audit_plan()`** on any LLM/parse error.
- `build_generation_prompt` (L73-215): three prompt variants — Builtin (tailor built-ins to spec), Merged (built-ins + user doc, user wins), Custom (user doc only). All require the LLM to return the exact JSON schema.
- `parse_audit_plan_response` (L221-260): extracts JSON, forces `mode` to the requested value, backfills `raw_principles`, rejects empty dimensions.

### 2.4 How auditPlan/auditMode/auditDocPath are wired end-to-end

- **Migration** `crates/db/migrations/20260508000000_add_audit_plan.sql`: adds `planning_draft.audit_plan TEXT`, `planning_draft.audit_mode TEXT NOT NULL DEFAULT 'builtin'`, `planning_draft.audit_doc_path TEXT`, and `workflow.audit_plan TEXT`.
- **Draft model** `crates/db/src/models/planning_draft.rs`: fields L36-41; `update_audit_plan` (L278-302); `delete_audit_doc` (L305-319, resets mode to `builtin`); `audit_mode` defaults `"builtin"` (L101).
- **Confirm endpoint** `crates/server/src/routes/planning_drafts.rs` `confirm_draft` (L286-390): decides mode from doc presence + `retain_builtin` (L306-313 → no doc=Builtin, doc+retain=Merged, doc+!retain=Custom), reads doc from disk (L316-332, dir = `utils::cache_dir()/audit_docs`), builds LLM client (draft model → Claude Code native fallback), calls `generate_audit_plan`, serializes JSON, `update_audit_plan`, then `set_confirmed`.
- **Upload/delete doc** `upload_audit_doc` (L1048-1138) / `delete_audit_doc` (L1140-1188): multipart, 10MB limit (L135), allowed ext `md/txt/pdf/docx` (L1041), stored at `audit_docs/{draft_id}/{filename}`. Blocked once draft is confirmed/materialized/cancelled.
- **Materialize** `materialize_draft` (L904-1034): copies `draft.audit_plan` → `workflow.audit_plan` (L984). Audit mode/doc are NOT copied — only the resolved plan JSON.
- **Orchestrator consumption** `crates/services/src/services/orchestrator/agent.rs`:
  - `run_acceptance_review` (L5360-5444): loads `workflow.audit_plan` JSON → `AuditPlan` (L5386-5393, fallback `fallback_default_audit_plan()` = empty raw_principles ⇒ legacy binary path).
  - `build_scoring_review_prompt` (L5306-5356): injects `audit_plan.raw_principles` into the LLM prompt; parses with `AuditScoreResult::parse`; threshold 90.
- **Frontend types** `frontend/src/lib/api.ts`: `PlanningDraftResponse.auditPlan: string|null`, `auditMode: 'builtin'|'merged'|'custom'`, `auditDocPath: string|null` (L519-521); `planningDraftsApi.confirm(draftId, retainBuiltin)` (L589), `materialize` (L597), `uploadAuditDoc` (L605, **BUG: appends form field `'audit_doc'`** but backend `upload_audit_doc` only accepts field named **`'file'`** — `planning_drafts.rs` L1072), `deleteAuditDoc` (L619).
- **Frontend UI** `frontend/src/components/ui-new/containers/AuditDocPanel.tsx`: collapsible panel for upload/delete + a "retain built-in" checkbox. **It does NOT display or edit any rules/criteria** — it only manages a single uploaded doc. `usePlanningDraft.ts` provides `useUploadAuditDoc`/`useDeleteAuditDoc`.

## 3. DB / persistence picture

- `planning_draft` table: base `crates/db/migrations/20260307200000_add_planning_draft.sql` (+ feishu sync migrations); audit columns from `20260508000000_add_audit_plan.sql`.
- `workflow.audit_plan`: same audit migration.
- Quality-gate persistence (read-only audit trail, NOT rules): `quality_run` + `quality_issue` (`20260312140000_create_quality_tables.sql`); `quality_policy_snapshot` (`20260313100000`) stores the **`config_yaml` text per run** — the closest existing precedent for persisting a gate policy in the DB, and a strong model for a per-project rules table.
- `20260312130000_create_quality_gates.sql` is intentionally **empty** (superseded).
- ts-rs: only `QualityRunSummary` / `QualityRunDetail` are exported (`crates/server/src/bin/generate_types.rs` L231-232). **Neither `QualityGateConfig`/`ConditionConfig`/`MetricKey` nor `AuditPlan` has a generated TS type** — a UI editor would need new ts-rs derives or hand-written TS interfaces.

## 4. How rules reach a running terminal/gate today (thread for custom rules)

`crates/services/src/services/orchestrator/agent.rs`:
- **Terminal gate** (L2748-2929): per checkpoint commit. Computes merge-base + changed files, `ensure_js_deps_installed_for_gate`, then `QualityEngine::from_project(wd)` (L2815) + `run_with_scope(wd, Terminal, scope)` (L2817). Persists `quality_run` + `quality_issue` rows + `providers_json`/`decision_json`. Failure → `to_fix_instructions()` fed back to the terminal.
- **Branch gate** (L7801-7816) and **Repo gate** (L7820-7826): `QualityScope::TaskBranch` / `RepoFinal`, via the same `run_gate` helper (L7850-8009) using `QualityEngine::from_project(project_root)` (L7880).

**Key wiring fact:** the gate always re-derives its rules from `QualityEngine::from_project(working_dir)` — i.e. **from the project repo's `quality/quality-gate.yaml`** (or the bundled default). To thread *DB-backed per-project custom rules* into the terminal, the cleanest injection point is `QualityGateConfig::load_from_project` (add a DB-priority branch) OR a new `QualityEngine::from_config(config, root)` call at these three sites passing a config resolved from the DB.

## 5. Concrete data-model recommendation for the new feature

If feature = editable **gate rules** (System A — the literal "quality-gate rules"):
1. **Reuse `ConditionConfig` verbatim** as the wire/edit shape (`metric` + `operator` + `threshold`); reuse `GateDefinition` for the per-level grouping and `ProvidersConfig` for toggles. No struct changes needed.
2. **Built-ins to display** = parse `BUNDLED_CENTRAL_POLICY` (or read `quality/quality-gate.yaml`) → the three `GateDefinition`s. Expose a read endpoint that returns the default config so the popup can seed.
3. **Persist per-project overrides** in a NEW table, e.g. `project_quality_policy(project_id PK/FK, config_yaml TEXT, updated_at)` — mirrors `quality_policy_snapshot.config_yaml`. Store either full YAML or a JSON diff.
4. **Load order**: add DB lookup as priority 0 in `QualityGateConfig::load_from_project` (keyed by project), falling through to repo file → bundled.
5. **API**: add CRUD under the existing `quality_routes()` (`crates/server/src/routes/quality.rs`, currently read-only — runs/issues only). New: `GET/PUT /api/projects/{id}/quality-policy` (+ a `GET .../quality-policy/default`).
6. **TS types**: add `TS` derive to `QualityGateConfig`/`GateDefinition`/`ConditionConfig`/`MetricKey` and register in `generate_types.rs`, OR hand-write interfaces in `api.ts`.
7. **Constraint to surface in UI**: `MetricKey` is a closed 44-variant enum — the "add rule" UI must be a **picker over existing metrics**, not free-text metric creation. Operator is GT/LT only. One condition per metric per gate (equality-by-metric).

### Gaps where the editable-rules feature has NO existing backing (mark UNAVAILABLE)
- No DB table for per-project gate rules (only `quality_policy_snapshot`, which is a per-run audit copy, not an editable source).
- No API to read/write `QualityGateConfig` (quality routes are read-only).
- No load path that consults the DB for gate rules (engine reads repo file / bundled only).
- No ts-rs/TS type for the gate config or its conditions/metrics.
- No UI to display or edit gate conditions (`AuditDocPanel.tsx` only uploads/deletes one audit doc; it never lists rules).
- `MetricKey` cannot be extended at runtime — user-defined brand-new metrics are unavailable without Rust changes.
- (Confirm-time popup) the audit plan is LLM-generated free text — there is no structured per-criterion editor for `AuditPlan.dimensions[].criteria[]`.
- (Bug, not a gap) `api.ts uploadAuditDoc` sends form field `'audit_doc'` but the backend expects `'file'` — the upload path is currently broken.
