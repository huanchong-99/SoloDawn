# V1.0 双脑独立架构诊断：跷跷板效应根因与统一修复方案

> **诊断时间**: 2026-04-24
> **副脑 (Claude Opus 4.6)**: 独立代码探索 + 数据驱动分析
> **主脑 (Codex GPT-5.5, xhigh)**: 独立架构审计 + 修复方案设计
> **核心问题**: 修复 A 任务时 B 任务退化，修复 B 时 A 退化 — 跷跷板效应

## 二、副脑独立分析 (Claude Opus 4.6)

### 2.1 根因诊断：统一管道处理分裂任务

**核心发现**: 系统对两种根本不同的任务类型使用完全相同的管道。10 项修复全部是"统一应用"，优化了一端必然损害另一端。

#### 分裂点 1: QUALITY_REQUIREMENTS_SUFFIX — 一刀切指令膨胀

**文件**: `crates/services/src/services/orchestrator/constants.rs:95-150`

每个终端指令末尾追加 ~2500 字符的质量要求（5 大类 + 自检清单），通过 `dispatch_terminal()` 的唯一出口强制注入：

```rust
// agent.rs:5130 — 所有终端指令都经过这里
let instruction = format!("{instruction}{QUALITY_REQUIREMENTS_SUFFIX}");
```

**对从零任务**: 这些要求是必要的（Docker、CI、README、测试 — 全部需要从头创建）
**对既有代码任务**: 这些要求是噪音甚至有害：
- Task 2 (Hoppscotch): 已有 Docker/CI/README，终端浪费精力在"创建 Dockerfile"上
- Task 4 (重构): 终端被要求"创建 CI pipeline"而非专注重构本身
- Task 6 (安全+监控): 终端分散注意力在"README 有 setup 说明"上而非安全加固

**证据**: Task 4 R3 失分原因 "Prettier + Husky + lint-staged 完全缺失" — 终端在满足通用质量要求上耗尽了精力，而非完成任务特定需求。

#### 分裂点 2: 质量门变更文件范围不一致

**验收审查** (已修复): 使用 `compute_merge_base()` 做全分支 diff

```rust
// agent.rs:7514-7517 — fetch_terminal_completion_context
let base_ref = match (task_branch, target_branch) {
    (Some(tb), Some(tgt)) => compute_merge_base(working_dir, tgt, tb).await,
    _ => None,
};
```

**质量门** (未修复): 使用 `commit~1` 做单次 commit diff

```rust
// agent.rs:2049 — 质量门变更文件收集
let base = format!("{hash}~1");
collect_changed_files_for_quality_gate(wd, &base, hash).await
```

**更严重的问题**: 大部分质量 Provider 完全忽略 `changed_files` 参数：

| Provider | 是否使用 changed_files | 影响 |
|----------|----------------------|------|
| builtin_common | ❌ `_changed_files` | 扫描全项目 |
| builtin_rust | ❌ `_changed_files` | 扫描全项目 |
| completeness | ❌ `_changed_files` | 扫描全项目 |
| security | ❌ `_changed_files` | 扫描全项目 |
| repo | ❌ `_changed_files` | 扫描全项目 |
| sonar | ❌ `_changed_files` | 扫描全项目 |
| rust_analyzer | ❌ `_changed_files` | 扫描全项目 |
| builtin_frontend | ✅ 使用 | 仅扫描变更文件 |
| frontend | ✅ 使用 | 仅扫描变更文件 |

**对既有代码任务的影响**: 质量门报告整个现有项目的问题，终端被要求修复不是自己引入的问题。质量门 enforce 模式阻断终端完成，触发无限修复循环。

**证据**: Task 6 R3 "npm audit 12 漏洞未修复" — 这些漏洞是既有项目的依赖问题，不是 Terminal 的安全监控工作引入的。

#### 分裂点 3: Foundation 上下文对既有代码任务的污染

```rust
// agent.rs:5885-5886 — auto_dispatch_initial_tasks
let fctx = if task.order_index > 0 { foundation_ctx.as_deref() } else { None };
let goal = if task.order_index > 0 { initial_goal.as_deref() } else { None };
```

`build_foundation_context()` 使用 `git ls-tree -r --name-only HEAD` 获取文件树。对既有代码任务：
- 既有代码库可能有数千文件，`take(200)` 后仍然是巨大的文件列表
- 指令说 "Build upon this existing structure. Do NOT recreate files that already exist" — 这对既有代码任务毫无意义（它本来就不会重新创建）
- 消耗宝贵的指令空间，挤掉真正重要的任务描述

**对从零任务**: 这是必要的（功能终端需要知道 Foundation 创建了什么）
**对既有代码任务**: 纯噪音，浪费指令空间

#### 分裂点 4: 验收审查 prompt 无差异化

```rust
// agent.rs:3907-3932 — build_acceptance_review_prompt
"You are a code reviewer verifying whether a task's requirements have been met \
 by reading the actual source code."
```

同一个审查 prompt 用于两种完全不同的验收标准：
- **从零任务**: 应该问 "是否实现了完整的功能集？"
- **既有代码任务**: 应该问 "是否正确完成了特定修改且未引入回归？"

当前 prompt 偏向从零任务（"review ALL requirements"），对既有代码任务过于严格（判定"需求未满足"时可能因为该需求原本就在既有代码中已实现）。

#### 分裂点 5: 指令空间预算无管理

`build_task_instruction()` (agent.rs:5636-5753) 的输出由以下部分组成：

| 组件 | 最大字符数 | 从零任务需要 | 既有代码任务需要 |
|------|-----------|------------|--------------|
| initial_goal | 8,000 | ✅ 必要 | ⚠️ 可选（可精简） |
| foundation_context | ~4,000 (200行) | ✅ 必要 | ❌ 噪音 |
| sibling_tasks | ~2,000 | ✅ 必要 | ⚠️ 视情况 |
| task_description | 4,000 (多终端) / 无限 | ✅ 必要 | ✅ 必要 |
| role_description | 无限 | ✅ | ✅ |
| commit_metadata | ~300 | ✅ | ✅ |
| prev_context | ~2,800 | ✅ | ✅ |
| QUALITY_SUFFIX | ~2,500 | ✅ 必要 | ❌ 大部分噪音 |

**从零任务总计**: ~24,000 字符 — AI CLI 可能截断或忽略后半部分
**既有代码任务总计**: ~24,000 字符 — 其中 ~14,000 是噪音

无论哪种类型，指令都太长。但损害方式不同：
- 从零任务: 质量要求被截断 → 缺少测试/Docker/CI
- 既有代码任务: 真正的任务描述被噪音稀释 → 缺少特定功能

### 2.2 副脑结论

**根本原因不是单个 bug，而是架构级缺陷：系统没有"任务类型"概念。** 所有修复都是统一应用的。每次修复本质上是在"从零"和"既有"两个方向之间移动一个全局旋钮。不可能通过调一个旋钮同时满足两个方向。

## 三、主脑独立分析 (Codex GPT-5.5, xhigh)

> **模型**: GPT-5.5, effort=xhigh
> **分析方式**: 独立代码探索，未共享副脑发现

### 3.1 根因分析

**跷跷板的根因是一条共享编排路径同时服务两种不同的工作流原型，却没有持久化的模式区分。**

`build_initial_planning_prompt()` 让 LLM 推断 "FROM-SCRATCH" vs "MODIFY-EXISTING"，但状态模型只存储 `foundation_phase_only`，不是真正的工作流类型（`agent.rs:508`, `state.rs:82-90`）。

**最强耦合 Bug**: `order_index > 0` 被用作"非 Foundation"的判断。在 `dispatch_next_terminal()` 和 `auto_dispatch_initial_tasks()` 中，原始需求和 Foundation 上下文只传递给 `task.order_index > 0` 的任务（`agent.rs:3111-3125`, `agent.rs:5885-5888`）。**因此既有代码工作流的第一个任务（往往 `order_index == 0`）被剥夺了原始需求，即使它不是 Foundation 任务。** 这正是从零修复导致既有代码任务退化的机制。

另一个共享杠杆是全局质量后缀。`dispatch_terminal()` 将 `QUALITY_REQUIREMENTS_SUFFIX` 追加到每条终端指令（`agent.rs:5130`），要求每个终端产出 README、Docker、CI、测试（`constants.rs:95-150`）。加强则既有任务退化，放宽则从零任务退化。

### 3.2 系统级问题（5 项）

#### Issue 1: 隐式工作流模式

工作流模式由 LLM 在 prompt 文本中推断（`agent.rs:508`），然后退化为 `foundation_phase_only` 和 `task_count <= 1` 守卫（`agent.rs:4604-4620`）。Foundation 行为泄漏到既有代码工作流；第一个既有任务丢失 goal 上下文；单任务既有工作流无法正常完成。

#### Issue 2: 验收审查在终结化之后运行

终端在 `try_finalize_terminal()`（`agent.rs:1509`）被终结并关闭后，任务状态在审查前就已更新（`agent.rs:1574-1589`），然后审查拒绝时尝试向已完成/已关闭的终端发送修复指令（`agent.rs:1636-1671`）。**被拒绝的工作可能已经被标记为 completed，使修复送达不可靠且状态不一致。**

#### Issue 3: 只有 Terminal Gate 实际执行

质量引擎始终以 `QualityGateLevel::Terminal` 运行（`agent.rs:2092-2097`）。YAML 定义了 Branch/Repo 级质量门（`quality/quality-gate.yaml:63-173`），包括覆盖率和安全规则，但任务完成和合并流程从未触发它们（`agent.rs:6059-6211`）。**终端 commit 可能通过，但分支级集成、覆盖率、仓库安全和最终完整性检查从未执行。**

#### Issue 4: Provider 适用性和度量失败不协调

引擎通过 `applicable_metrics()` 过滤条件（`engine.rs:201-210`），但 trait 默认返回所有支持的度量（`provider/mod.rs:158-164`）。`RustProvider` 未覆盖此方法，对无 `Cargo.toml` 的项目发出 `-1` 哨兵值（`rust_analyzer.rs:230-320`）；求值器将 `-1` 转为 error（`evaluator.rs:50-63`）；代理将 `blocking_issues == 0` 的 enforce 失败视为度量收集失败并自动通过（`agent.rs:2369-2374`）。**跨栈质量门既可能假阻断又可能 fail-open。**

#### Issue 5: 质量信号遗漏必要 Artifacts

验收审查只包含选定的源码/配置扩展名，排除了 README、`.env.example`、`.gitignore` 和 CI 文件（`agent.rs:7666-7681`），而 prompt 却要求"只根据展示的代码判断"（`agent.rs:3931-3933`）。`BuiltinCommonProvider` 声称是语言无关检查但只扫描 Rust/TS/JS 文件（`builtin_common.rs:19-27`），尽管 weak-default 规则针对 Docker/配置密钥（`weak_default_detection.rs:46-71`）。**文档、部署、CI、env 和配置质量被要求但从未被可靠验证。**

### 3.3 主脑修复方案

#### 修复 1: 显式工作流原型
- 引入持久化的 `WorkflowPlanKind::{FromScratch, ExistingCodebase}`
- 从确定性仓库发现（而非 LLM 推断）确定
- 用于 `build_initial_planning_prompt()`、`SetWorkflowPlanningComplete`、`CompleteWorkflow`、`dispatch_next_terminal()`、`auto_dispatch_initial_tasks()`
- **预期效果**: Foundation 分阶段只适用于从零工作流；所有既有代码任务都收到 `initial_goal`

#### 修复 2: 范围感知质量契约
- 用按 `{workflow_kind, task_role, terminal_role}` 生成的契约替换全局后缀
- Foundation 要求脚手架/文档/CI/Docker；既有代码任务要求保持架构、针对性测试、不做无关基础设施变更
- **预期效果**: 从零质量保持高水平，不导致既有代码范围蔓延

#### 修复 3: 审查在完成之前
- 在 `try_finalize_terminal()` 和任务状态更新之前运行验收审查
- 拒绝时保持终端 working 或先重新启动，不标记任务完成
- agent-planned 模式下审查 LLM 失败时重试或标记为 review_pending，而非自动批准
- 审查包含 README、`.env.example`、`.gitignore`、`.github/workflows/*`、Dockerfiles、compose 文件
- **预期效果**: 被拒绝的工作不会进入 completed 状态

#### 修复 4: 实施 Branch/Repo 质量门 + 修复适用性
- 任务完成时运行 Branch gate，合并/完成前运行 Repo gate
- `RustProvider` 添加 `applicable_metrics()` 覆盖，无 Rust targets 时返回空
- 将度量 Error 无 issues 的情况转为合成阻断 `QualityIssue`
- **预期效果**: 无跨栈假信号，度量失败不再静默 fail-open

#### 修复 5: 扫描必要 Artifacts
- secret/weak-default 规则扫描文本/配置 artifacts
- 保持复杂度/重复度检查仅针对源码
- 仅在工作流类型要求时添加 README/CI/Docker/`.env.example` 完整性检查
- **预期效果**: 质量要求变为可验证的，而非仅靠 prompt

### 3.4 主脑架构洞察

**根本设计缺陷**: 编排策略、任务规划、终端分发、质量执行和完成状态全部耦合在同一个终端完成事件上。`handle_terminal_completed()` 同时是质量门路由器、验收审查器、任务状态修改器、下一终端分发器和工作流完成触发器（`agent.rs:1330-1744`）。这使得每项修复默认是全局的。

系统需要一个**显式工作流策略层**: 分类工作流类型一次，从该策略派生任务/终端契约，让质量门在终端、分支和仓库边界独立运行。今天 "Foundation" 不是一等公民；它从 `order_index`、`task_count` 和 prompt 文本推断。

### 3.5 主脑优先级排序

1. 添加显式工作流原型，移除 `order_index > 0` 作为阶段检测 — **直接解决跷跷板**
2. 将验收审查移到终结化之前 — 修复假完成和不可靠的拒绝修复
3. 修复 Provider 适用性 + 合成质量门问题 — 防止 fail-open/fail-closed 振荡
4. 在合并/完成前添加 Branch/Repo gate 执行 — 封堵集成/安全/覆盖率缺口
5. 让质量后缀和 artifact 扫描范围感知 — 保持从零严格性但不强制既有代码变更

## 五、统一修复方案

### 设计原则

**核心理念**: 引入"任务类型分类"，让管道的每个环节根据任务类型自适应。不是替换现有修复，而是在现有修复基础上增加维度。

### Fix A: 显式工作流原型 (Workflow Archetype)

> 双脑共识：这是解决跷跷板的**第一优先级修复**

**位置**: `agent.rs`, `state.rs`

**变更**: 
1. 引入显式工作流原型，从**确定性仓库发现**（而非 LLM 推断）确定：

```rust
#[derive(Clone, Copy, Debug, PartialEq)]
enum WorkflowArchetype {
    FromScratch,       // 从零创建：空仓库或空目录
    ExistingCodebase,  // 既有代码：仓库已有源文件
}

/// 确定性推断：检查工作目录是否已有源文件
async fn detect_workflow_archetype(working_dir: &Path) -> WorkflowArchetype {
    let has_existing_source = tokio::process::Command::new("git")
        .args(["ls-files", "--cached", "*.rs", "*.ts", "*.tsx", "*.js", "*.py", "*.go"])
        .current_dir(working_dir)
        .output()
        .await
        .map(|o| !String::from_utf8_lossy(&o.stdout).trim().is_empty())
        .unwrap_or(false);
    
    if has_existing_source {
        WorkflowArchetype::ExistingCodebase
    } else {
        WorkflowArchetype::FromScratch
    }
}
```

2. 在 `OrchestratorState` 中持久化：`pub workflow_archetype: WorkflowArchetype`
3. 用于所有下游决策（替换 `order_index > 0` 和 `foundation_phase_only` 推断）
4. **关键修复**: `auto_dispatch_initial_tasks()` 中不再用 `task.order_index > 0` 判断是否传递 `initial_goal`：

```rust
// BEFORE (有 bug):
let goal = if task.order_index > 0 { initial_goal.as_deref() } else { None };

// AFTER (根据原型):
let goal = match archetype {
```

## 十一、主脑追加独立审计：为什么越修越降分

### 11.1 递减机制：修复在调全局旋钮，而不是固定系统边界

本轮独立审计确认，分数持续下降并不是因为“同一个 bug 反复没修好”，而是因为每一轮修复都在共享路径上调节全局行为，缺少稳定的任务类型边界、质量债务边界和完成状态边界。`handle_terminal_completed` 同时负责终端完成、质量门禁、任务状态更新、终端关闭、验收审查和后续调度，单个函数覆盖整个终端生命周期（`crates/services/src/services/orchestrator/agent.rs:1330`-`1744`）。因此任何为 from-zero 任务增加严格度的修复，都会同步作用到 existing-codebase 任务；任何为 existing-codebase 降低噪声的修复，也会同步削弱 from-zero 任务的完整性约束。

最明显的共享旋钮是全局 prompt 合约。所有终端都会在 `dispatch_terminal` 阶段追加同一个 `QUALITY_REQUIREMENTS_SUFFIX`（`crates/services/src/services/orchestrator/agent.rs:5121`-`5135`），该 suffix 同时要求 README、Dockerfile、docker-compose、CI、`.gitignore`、测试、格式化和安全检查（`crates/services/src/services/orchestrator/constants.rs:95`-`158`）。这套要求适合 from-zero 项目收敛交付物，却会把 existing-codebase 的小范围修改推向无关基础设施变更；反过来，如果为 existing-codebase 放松这些要求，Task 1/5 这类从零任务就会缺少可验证的交付物闭环。

系统当前也没有持久化的工作流 archetype。初始规划 prompt 要求 LLM 判断 `FROM-SCRATCH` 或 `MODIFY-EXISTING`（`crates/services/src/services/orchestrator/agent.rs:497`-`511`），但状态层只有 `workflow_planning_complete` 和 `foundation_phase_only` 这类流程标志（`crates/services/src/services/orchestrator/state.rs:49`-`85`），没有不可变的 `WorkflowArchetype` / `QualityProfile`。这导致后续调度、质量门禁、验收审查都无法基于“本工作流到底是 greenfield 还是 existing-codebase”执行不同的不变量，只能依靠 `order_index`、任务数量和 prompt 文本做间接推断。

### 11.2 需要补充的关键缺口

1. **没有质量债务边界**

   证据：终端门禁的 diff 基准仍然是当前提交的父提交 `hash~1`（`crates/services/src/services/orchestrator/agent.rs:2049`-`2052`），而多个 provider 又忽略 `changed_files`，例如 completeness provider 全项目扫描源码和测试（`crates/quality/src/provider/completeness.rs:58`-`120`），builtin common provider 全项目扫描 Rust/TS/JS 文件（`crates/quality/src/provider/builtin_common.rs:47`-`52`），security provider 也不按变更文件收敛（`crates/quality/src/provider/security.rs:31`）。影响是 existing-codebase 任务会被历史技术债阻塞，from-zero 任务却确实需要全量约束；两类任务被同一套 repo 级检查拉扯，形成跷跷板。

2. **质量门禁存在 fail-open 空洞**

   证据：Rust provider 在没有 `Cargo.toml` 时写入 `-1` 指标但不生成 `QualityIssue`（`crates/quality/src/provider/rust_analyzer.rs:230`-`320`）， evaluator 将 `-1` 视为 ERROR（`crates/quality/src/gate/evaluator.rs:50`-`63`），但 `QualityReport::aggregate` 只统计 provider issues（`crates/quality/src/report.rs:35`-`45`）。随后 orchestrator 遇到 `passed=false,total_issues=0,blocking_issues=0` 会把它当成指标采集失败并提升为 completed（`crates/services/src/services/orchestrator/agent.rs:2369`-`2374`）。影响是同一套质量门禁在某些项目上“看似严格但不可操作”，在另一些项目上“看似通过但实际未验证”，分数会随项目栈和工具可用性漂移。

3. **验收审查不是硬状态转换**

   证据：`handle_terminal_completed` 先调用 `try_finalize_terminal` 并执行终端关闭（`crates/services/src/services/orchestrator/agent.rs:1509`-`1517`），之后才运行 acceptance review（`crates/services/src/services/orchestrator/agent.rs:1619`-`1671`）。`run_acceptance_review` 在没有可读变更内容时直接批准（`crates/services/src/services/orchestrator/agent.rs:3952`-`3953`），LLM 不可用时也直接批准（`crates/services/src/services/orchestrator/agent.rs:3988`），JSON 缺失 verdict 时解析层默认 approved（`crates/services/src/services/orchestrator/types.rs:473`-`520`）。影响是验收审查无法稳定阻止错误完成，拒绝结果还可能发生在终端已经被标记完成之后。

4. **交付物合约停留在 prompt，没有对应 verifier**

   证据：全局 suffix 要求 README、Docker、compose、CI、环境样例等交付物（`crates/services/src/services/orchestrator/constants.rs:95`-`158`），但验收读取变更内容只包含有限源码和配置扩展名（`crates/services/src/services/orchestrator/agent.rs:7666`-`7670`），不覆盖 `md`、`.env.example`、`.github/workflows`，且 builtin common provider 只收集 Rust/TS/JS 文件（`crates/quality/src/provider/builtin_common.rs:20`-`23`），即使 weak-default 规则本身支持 yaml/env/json/toml（`crates/quality/src/rules/common/weak_default_detection.rs:109`-`118`）。影响是 from-zero 任务的关键交付物可能只被“要求”而未被验证，existing-codebase 任务却仍承受这些无关要求带来的 prompt 噪声。

5. **只运行 Terminal gate，没有真正的分阶段交付门**

   证据：质量配置定义了 Terminal、Branch、Repo 三层门禁（`crates/quality/quality-gate.yaml:17`-`173`），但 orchestrator 实际调用 `engine.run(wd, QualityGateLevel::Terminal, ...)`（`crates/services/src/services/orchestrator/agent.rs:2092`-`2097`）。合并与完成路径主要依赖任务状态和自动同步（`crates/services/src/services/orchestrator/agent.rs:6059`-`6211`），没有在 branch/repo 层重新校验集成质量。影响是终端局部修复不能保证最终工作流质量，尤其是并行任务合并后会出现“每个终端都像通过了，但整体评分下降”。

### 11.3 系统性架构整改方案：同时约束六个任务的版本

1. **引入持久化 `WorkflowArchetype`、`QualityProfile`、`DebtPolicy`**

   修改位置：`crates/services/src/services/orchestrator/state.rs` 新增不可变工作流类型字段；`crates/services/src/services/orchestrator/agent.rs::build_initial_planning_prompt` 只负责生成计划候选，不再作为唯一分类来源；新增确定性分类函数，在 workflow 初始化时根据仓库文件数量、manifest、git history、任务文本和 foundation 需要写入状态。

   预期结果：from-zero、foundation-first、existing-codebase 不再共享同一组 prompt/质量/完成规则。后续 `auto_dispatch_initial_tasks`、`dispatch_next_terminal`、`CompleteWorkflow`、`build_task_instruction` 都读取该状态，而不是用 `order_index > 0` 或任务数量猜测上下文。

2. **把 `changed_files` 升级为 `QualityScope`，并建立质量债务基线**

   修改位置：`crates/services/src/services/orchestrator/agent.rs::handle_checkpoint_quality_gate` 不再使用 `hash~1` 作为唯一基准；按 archetype 传入 `IntroducedOnly { merge_base, head, files }`、`BranchComplete` 或 `RepoFinal`。`crates/quality/src/provider/mod.rs` 将 provider 输入从裸 `changed_files` 升级为 scope；completeness、security、builtin_common、rust/frontend provider 必须实现 scope-aware 行为。

   预期结果：existing-codebase 的终端门禁只阻塞“本次引入或本任务承诺修复”的问题，历史债务进入 baseline；from-zero 任务仍可在 Branch/Repo gate 中全量阻塞。这样严格度不是降低，而是归位到正确边界。

3. **禁止质量门禁 `passed=false,total=0,blocking=0` 自动完成**

   修改位置：`crates/quality/src/engine.rs` 在 `ConditionEvaluator::evaluate_all` 后，为每个 ERROR 结果生成可操作的 synthetic `QualityIssue`；`crates/quality/src/provider/rust_analyzer.rs` 实现 `applicable_metrics`，没有 Rust target 时不声明 Rust 指标；`crates/services/src/services/orchestrator/agent.rs::handle_quality_gate_result` 删除 0 issue 的自动 promote 分支，改为 blocking infra issue 或等待人工/自动修复策略。

   预期结果：门禁失败一定有原因、原因一定可修复或可归类为环境阻塞；不会再出现某些任务因工具/指标不匹配静默放行，另一些任务被相同门禁错误阻塞的漂移。

4. **将 acceptance review 前移为硬门禁状态**

   修改位置：拆分 `crates/services/src/services/orchestrator/agent.rs::handle_terminal_completed`，顺序调整为：终端报告完成 -> Terminal quality pass -> acceptance review pass -> `try_finalize_terminal` -> 关闭终端 -> 标记 task completed。`TASK_STATUS_REVIEW_PENDING`（`crates/services/src/services/orchestrator/constants.rs:51`）应变成真实状态，并修正为可读的 `review_pending` 值；`run_acceptance_review` 在无内容或 LLM 不可用时返回 pending/blocking，而不是 approved。

   预期结果：LLM 验收从“完成后的建议”变成“完成前的准入条件”。拒绝结果会回到仍可修复的终端/任务状态，避免完成状态污染后续调度。

5. **用策略化 `QualityContract` 替代全局 suffix**

   修改位置：`crates/services/src/services/orchestrator/constants.rs::QUALITY_REQUIREMENTS_SUFFIX` 拆分为 contract 模板；`crates/services/src/services/orchestrator/agent.rs::dispatch_terminal` 和 `build_task_instruction` 根据 `WorkflowArchetype + task_phase + terminal_role` 渲染不同合约。

   预期结果：greenfield foundation 终端必须产出项目骨架、README、env、Docker/CI 等交付物；greenfield feature 终端必须复用 foundation 并补测试；existing-codebase 终端默认最小变更、聚焦现有架构、只要求与任务相关的测试和文档。两类任务不再通过同一段 prompt 互相牵制。

6. **补齐 Branch/Repo final readiness gate**

   修改位置：`crates/services/src/services/orchestrator/agent.rs::auto_sync_workflow_completion`、`execute_auto_merge`、`trigger_merge` 和 `CompleteWorkflow` 分支，在最终完成/合并前运行 `QualityGateLevel::Branch` 和必要的 `QualityGateLevel::Repo`；同时新增 artifact readiness matrix，把 README、Docker、compose、CI、env、tests、security defaults 映射到确定性 verifier。

   预期结果：终端级通过只代表局部可接受，最终评分维度由 branch/repo gate 统一兜底。并行开发时即使单个终端未发现集成问题，最终完成前也会被确定性门禁拦住。

### 11.4 新的不变量：防止再次跷跷板

1. existing-codebase 终端不得因未触碰的历史债务失败，除非任务明确要求清理该债务。
2. from-zero 工作流不得在 artifact readiness matrix 未满足时完成，除非存在显式 waiver 和审计记录。
3. 任何质量门禁 ERROR 必须转化为 blocking issue、环境阻塞或 pending 状态；不得自动 completed。
4. acceptance review 拒绝必须发生在 task completed 之前，并且拒绝后必须有可继续修复的执行上下文。
5. prompt 中出现的交付要求必须有对应 verifier；没有 verifier 的要求不能作为评分关键路径。

### 11.5 修订后的优先级

1. **P0：工作流类型和质量策略持久化**。先切断根耦合，否则后续所有修复仍会落在全局旋钮上。
2. **P0：验收审查前移为硬门禁**。这是防止“错误完成后再发现问题”的最短路径。
3. **P1：质量债务 baseline + provider scope-aware 改造**。这是 existing-codebase 不再被历史问题拖低、from-zero 不被放松的关键。
4. **P1：修复质量门禁 fail-open 空洞**。确保每次失败都可解释、可修复、可复现。
5. **P2：Branch/Repo readiness gate 和 artifact verifier**。把最终评分维度从 prompt 约束升级为确定性验收。

结论：要同时把六个任务推到 90+，不能继续追加单点补丁。必须把“任务类型、质量范围、债务基线、验收状态、最终交付物”五件事提升为一等架构对象。只有这些边界稳定后，from-zero 的完整性和 existing-codebase 的最小侵入性才不会再互相抵消。
