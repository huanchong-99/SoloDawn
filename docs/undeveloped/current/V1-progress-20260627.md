### task5 idx5 hung + §8 CPU-flat stall recovery 修复（2026-06-29 01:30）

**现象**：idx5 Final Integration Repair 运行 ~63min（5 个有价值集成 commit：anti-corruption layer f61b351、order event 12d5182、compose+CI 6a088b0、WebSocket test fa27161、admin render f982cd0），4 个 final-gate blocker 全解决（README 357L / 5 Dockerfile / ci.yml / 31 test 文件）。但 idx5 claude.exe **死锁**——冻结在 "Calculating" spinner，terminal_log 19.5min 零更新，**CPU Δ=0.000s/12s 绝对平坦**（对照活跃 claude Δ=470ms/12s）。系统无法自愈：stall sweep 每轮都走 `alive + quiet<STALL_ALIVE_HARD_CAP(3600s) → continue(保护)` 分支跳过它，要等满 1h hard-cap 才 force-complete。

**§8 根因**：stall detector 的保护分支无法区分**真 LLM 生成**(CPU 持续增长，TUI render loop + response processing 不停) vs **进程 hung/死锁**(CPU 平坦)。alive PID + 冻结 spinner + 平坦 CPU = hung，但被当作"生成中"保护满 1h。

**修复**(agent.rs 两处 Edit)：
1. 保护分支(`alive + quiet<hard_cap → continue`)前插 CPU-flat 检查：`is_terminal_process_cpu_flat` 返回 Some(true) → warn + `set_completed_if_unfinished` + continue（force-complete），否则 continue（保护）
2. 新增 `is_terminal_process_cpu_flat(pid) -> Option<bool>`：8s 窗口双采样 `read_process_cpu_time_ms`，Δ<50ms=平坦。None=保守当"非平坦"继续保护（无回归）
3. 新增 `read_process_cpu_time_ms`：Windows `wmic process where ProcessId=X get UserModeTime,KernelModeTime /format:csv`（100ns→/10000=ms）；Unix `/proc/pid/stat` utime+stime (field 11/12, ×10ms 假设 100Hz)

**验证**：`cargo check -p services` 通过(15.92s)；cargo-watch 自动重编译部署 **server PID 41576 @01:30:53**（旧 47908 替换）。实战信号 = backend.log `"flat CPU over sample window — hung/deadlocked, not generating; force-completing"`。阈值 50ms：实测 hung=0ms / 活跃=470ms / spinner 动画>50ms，安全分离。

**附带效应**：server 重启清理旧 hung claude.exe(50252) → idx5 被 re-dispatch（新 claude.exe 50864 活跃工作）。idx5 可能靠重启 re-dispatch 先收敛，也可能再 hang 触发修复。修复对**任务 4/3/2/6 的未来 hung 进程防御性生效**。
