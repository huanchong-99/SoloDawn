**鉴别：非 GLM provider 问题（§7）**：backend.log 04:22:11 + 04:27:02 有 `Anthropic-compatible LLM request starting`，无 429/timeout/event:error/empty-content。CPU 双采样推翻"等 LLM 响应"初判 —— **3 个 claude（14980/66340/56516）全 CPU flat（delta=0s/12s）**。

**真根因（claude 进程内部 hung）**：14980 有 476s CPU（启动后工作过：初始化+读文件+几次 LLM），但随后 **CPU flat + idle-prompt 残留 + 76min 无 commit + GLM 正常** = claude 进程**内部卡死**（alive PID 但不再读 PTY stdin）。stall detector 的 idle-prompt 分支（#N3 companion 修复，agent.rs:1750）对 idle-prompt 一律 fall-through re-dispatch —— 但 **re-dispatch 同一个 hung 进程是永久无效的**：注入的 instruction 永远不被消费，stall detector 每 quiet window（180s）再 re-dispatch，循环 76min+。区别于 #N3 companion 原始场景（fresh claude cold-start swallow，re-dispatch 可自愈）—— 本条是**已经工作过一段后卡死的 hung 进程**，re-dispatch 无法唤醒。

**修复 [G32-idle-flat-forcecomplete]**（agent.rs idle-prompt 分支）：加 CPU-flat 守卫。idle-prompt 命中后：
- `idle_cpu_flat = is_terminal_process_cpu_flat(pid) == Some(true)` → **force-complete**（`set_completed_if_unfinished`，kill stuck 进程），让 sweep re-dispatch **spawn FRESH claude**（fresh claude 有真实概率接收 instruction —— oid=2 的 coders 这样达 92）。
- CPU active → **保护**（continue，等 LLM 响应，不打断）。

关键：force-complete kill hung + spawn fresh 是唯一能恢复的路径（re-dispatch 同一 hung 进程无效）。G32 跳过无效 re-dispatch 循环，加速 spawn fresh。**不误杀活跃生成**：idle-prompt + CPU active 走 protect；只有 idle-prompt + CPU flat（真 hung）force-complete。活跃生成 CPU 持续增长，不会被误判 flat。

**诊断矩阵**：(1) `Live terminal at idle input prompt` 反复 + 同 terminal 无 commit + claude CPU flat（delta≈0）= 本缺陷；(2) GLM provider 有 LLM request 无 error = 非 §7 模型侧；(3) claude CPU 有累计（如 476s）但当前 flat = 工作过后 hung（非 fresh swallow）；(4) re-dispatch 12+ 次无 commit = re-dispatch 无效，需 force-complete。

**姊妹**：[[cpu-flat-stall-recovery]]（非-idle 分支 CPU-flat→force-complete，本条 idle 分支补齐）、[[stall-recovery-redispatch-race]]（反面：误判生成中）、[[claude-welcome-swallow-instruction]]（fresh swallow vs 本条工作后 hung）、[[arrowselect-claude-tui-flood]]（G31，本 wf 先修，释放 oid=2→92）。

### §8 根因 #19：orphaned completed-terminal 无 review，not_started 占位符阻塞 sweep 恢复（2026-06-30 04:57 修复）

**现象**：wf 6a2e3214 跑 7.5h，oid=0/oid=1 的 coder 终端（e561b1ec/67f4299b）于 ~00:00 完成 handoff commit（"Real-time Dashboard & Charts handoff" / "UI Navigation, Config & Collections Sel..."），但 **acceptance review 从未触发**（grep 全日志无 "Running acceptance review" / "scoring complete" 匹配这两个 task）。task 卡 `running` + score=null + verdict=null。sweep 跨 6 次重启（00:29/01:09/01:22/01:23/03:19/04:43）反复日志 `Skipping terminal e561b1ec for task 1daf7cc6 due to status completed` —— 永久跳过，永不补 review。

**为何 oid=2/4 有 review 而 oid=0/1 没有**：oid=2/4 两终端都 completed（双 coder handoff）→ `auto_complete_stalled_tasks` 的 `all_done=true` → 进入 synthetic-review 恢复块 → review 跑。oid=3/5 经正常终端完成路径（handle_terminal_completed → run_acceptance_review_gate）有 review。**oid=0/1 完成于 ~00:00（G29 review 修复 01:09 之前），正常完成路径未产 review；之后 sweep 无法恢复**，因为每个 task 有第二个 `not_started` 占位终端 → `all_done=false`（要求全 completed/failed）→ synthetic-review 恢复块（agent.rs 原 1302，gated 在 `if all_done` 内）永不执行。

**真根因**：`auto_complete_stalled_tasks`（agent.rs:1192）的 `all_done` 判定 `task_terminals.iter().all(|t| t.status=="completed"||t.status=="failed")` —— agent-planned feature task 通常有 1 个 coder 终端（completed 带 commit）+ 1 个 `not_started` follow-up 占位槽。占位槽使 `all_done=false`，卡住整个恢复路径：synthetic review 不跑 → score 永远 null → task 永卡 running → wf 永不完成。

**修复 [G33-orphan-completed-terminal-recovery]**（agent.rs `auto_complete_stalled_tasks` all_done 判定）：扩展 `all_done` —— 当所有终端都 inert（completed/failed/**not_started**）**且**至少一个 completed 终端带非空 commit_hash（coder 完成了真实工作）时，也判定为 done。这使 synthetic-review 恢复块（`is_agent_planned && verdict.is_none()` → run_acceptance_review_gate）能为 oid=0/1 补跑缺失的 review：approved→complete；rejected→re-drive。

**为何安全**：(1) 仅当无 `working`/`starting`/`waiting` 活跃终端时触发（不打断在跑的 coder）；(2) 要求至少一个 completed+commit 终端（纯 not_started 早期任务不会误触发 —— synthetic review 找 `.find(|t|t.status=="completed").or_else(first)`，无 completed 终端则无 commit → review 自然 reject/error，但 has_completed_commit 守卫已在 all_done 前置拦截此情况）；(3) not_started 是永不启动的占位槽，不携带未完成工作。

**诊断矩阵**：(1) task `running` + score=null + verdict=null + 有 completed 终端带 commit + 有 not_started 终端 = 本缺陷；(2) grep `Skipping terminal X for task Y due to status completed` 反复跨重启 = sweep 卡在恢复门外；(3) 该 task 全日志无 "Running acceptance review" = review 从未跑（区别于 review 跑了但失败的 #15）。

**姊妹**：[[review-not-invoked-event-ordering]]（PR#29，review 在 auto-complete sweep 跑 —— 本条是其遗漏场景：all_done 门把 orphan 挡在门外）、[[acceptance-review-history-bloat]] #15（review 跑了但 empty-content 失败 —— 本条 review 根本没跑）、[[orphan-pending-feature-tasks-race]] #8（pending 无终端 —— 本条 running 有 completed 终端无 review）。
