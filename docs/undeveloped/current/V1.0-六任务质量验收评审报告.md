## 根因分析

### 发现的两个架构缺陷

**缺陷 A: 编排器从未读过实际代码**

编排器决定"终端是否完成"时看到的是：
- Terminal 日志尾巴（50 行，2000 字符）
- `git diff --stat`（文件名 + 行数统计，不含内容）
- Commit message（终端自己写的）

**从未读过一行实际源码。** 等于让学生自己改卷子。空壳服务只要能编译、commit message 说"已完成"，编排器就信了。

**缺陷 B: Foundation-Only 工作流完成 Bug**

编排器创建 Foundation 任务后标记 `workflow_planning_complete=true`，Foundation 完成后所有任务都"完成"了 → 工作流自动结束，功能任务从未被创建。任务 5（40 分）直接受此影响。

## 已实施的架构修复

### 修复 1: LLM 验收审查 (commit `96e7f0cc4`)

**原则**: 用与人工审查相同的方法 — 读代码、理解需求、判断完整性。

终端完成时的新流程：
```
commit → 质量门(编译/lint/测试) → 安静窗口 → 验收审查(读代码+对比需求)
                                                ↓ REJECTED → 反馈→继续工作
                                                ↓ APPROVED → LLM完成决策 → done
```

三个新函数：
- `collect_changed_files_content()` — 用 `git show` 读取变更文件实际内容（最多 32KB），源文件优先
- `build_acceptance_review_prompt()` — 构造审查 prompt：任务需求 + 实际代码 + 终端输出
- `run_acceptance_review()` — 仅 agent-planned 模式触发，REJECTED 时复用质量门 PTY 注入通道

### 修复 2: Foundation-Only 完成 Bug (commit `41e6b037d`)

- `SetWorkflowPlanningComplete` 检测到仅 1 个任务时，推迟 `workflow_planning_complete=true`，进入 `foundation_phase_only` 模式
- Foundation 完成后注入重规划指令，强制创建功能任务
- Agent-planned 工作流在 LLM 不可用时拒绝 auto-complete
- `CompleteWorkflow` 拒绝仅 1 个已完成任务的 agent-planned 工作流

### 修复 3: 通用质量检查 (commit `41e6b037d`)

- `test_file_absence` — 项目有源文件但零测试文件时阻断（通用原则）
- `todo_density` — TODO/FIXME 密度超 5% 时阻断（通用原则）
- `WeakDefaultDetectionRule` — 检测 Docker Compose 弱密码、JS 默认回退值、弱 JWT secret
- Goal preview 从 2000 字符提升到 16000 字符

### 修复 4: 完成守卫 (commit `41e6b037d`)

- `CompleteWorkflow` 增加目标词覆盖检查
- Agent-planned 工作流 LLM 不可用时拒绝 auto-complete

## LLM 深度架构审计 (2026-04-21)

> **审计模型**: Claude Opus 4.6 (1M context)
> **审计方式**: 3 个独立子代理分析编排器核心代码，交叉验证
> **审计范围**: agent.rs, constants.rs, prompt_handler.rs, state.rs

### Q1: 为何既有代码库任务（4/3/6）止步 73-85 分？

**四个系统级瓶颈：**

1. **验收审查只看最后一次 commit** — `agent.rs:7257` 使用 `git diff commit~1 commit`，多次 commit 的终端审查只看到最后一次（往往是 README 或格式修复），遗漏核心实现代码
2. **32KB 代码上限** — `ACCEPTANCE_REVIEW_MAX_BYTES = 32_768`，大型重构轻松超限，超出部分被静默丢弃，审查 LLM 看不到完整代码
3. **多终端任务描述截断到 200 字符** — `build_task_instruction()` 中 `total_terminals > 1` 时截断到 200 字符，丢失关键需求细节
4. **35 条质量门规则全是静态分析** — 无一条验证"API 端点是否真正可用"或"测试是否有意义"

### Q2: 为何从零创建任务（1/5/2）仅能达到 D 级？

**根因全部是系统级缺陷：**

1. **功能终端对 Foundation 输出零可见性** — `fetch_previous_terminal_context()` 只查同 task_id 的终端。Foundation 是独立 task，其输出（项目骨架、目录结构、依赖配置）对功能终端不可见。功能终端面对空目录只能产出 stub
2. **`prev_context` 只查同 task 的前一个终端** — 无法跨 task 访问。Foundation 代码已在 git 中，但终端不知道它创建了什么
3. **`PromptKind::Input` 无条件升级为 `ask_user`** — LLM 决策路径标记 `// TODO: Integrate actual LLM call` 但从未实现。终端遇到自由输入即永久卡死
4. **`initial_goal` 从未传递给终端** — 原始用户需求只存在于编排 LLM 的 system prompt 中；终端只看到 LLM 生成的任务摘要（可能被截断到 200 字符）

### Q3: 三层质量门 + LLM 审计哪里失效？

| 组件 | 状态 | 问题 |
|------|------|------|
| Terminal Gate | 部分有效 | 捕获编译/lint 错误，但无法检测 `expect(true).toBe(true)` 式空测试 |
| Branch Gate | 对从零项目无效 | 无覆盖率工具的空项目度量指标无法计算 |
| LLM Acceptance Review | 架构正确，执行残缺 | 只检查最后一次 commit + 32KB 限制 + LLM 失败时自动 pass + 空 diff 自动 pass |
| Quality Rules | 表层检查 | 35 条规则检查样式/复杂度；无一条验证功能正确性 |

**一句话总结**: 系统有验收审查机制，但被三个限制残废（单 commit 视角、32KB 上限、失败自动 pass）。修复这三点 + 传递 `initial_goal` + 启用跨 task 上下文共享，预期可将所有任务提升至 85-95 分。

## 第三次架构修复 (2026-04-21 ~ 2026-04-22)

> 在 Round 3/4 测试中发现 3 个新的系统级 bug，阻止 Foundation→Feature 过渡

### Bug 1: JSON 提取抓取多个 JSON 块 (commit `487b05078`)

**根因**: `extract_json_from_mixed_response()` 使用 `rfind(']')` 查找最后一个 `]`。当 LLM 返回两个 JSON 块（create_task + create_terminal）时，提取到无效 JSON（包含中间的散文文本）。

**修复**: 替换为深度感知括号匹配，正确提取第一个完整 JSON 块。

### Bug 2: Foundation→Feature 过渡死锁 (commit `52bdc1c7f`)

**根因**: Foundation 任务被 `auto_complete_stalled_tasks` 标记为 "completed"，但该路径不检查 `foundation_phase_only` 标志。正常路径（`handle_terminal_completed`）才会触发第二轮规划。R8-B3 guard 阻止终端完成（质量门未通过），但任务已完成→终端永远无法完成→过渡永远不触发。

**修复**: 
1. R8-B3 guard 豁免：当父任务已 completed 时，允许终端完成（避免死循环）
2. 卡住检测：`auto_complete_stalled_tasks` 末尾检测 foundation_phase_only + Foundation 已完成 + 无功能任务 → 触发合成完成事件调用 LLM 创建功能任务

### Bug 3: 质量门 enforce 停滞循环 (commit `5402a695f`)

**根因**: 质量门 secret-detection 对 `dist/` 编译产物持续报告阻断问题。终端修复源码后 `dist/` 未重建，阻断计数不变。每次 commit 重置计数器，stall recovery 永远不触发 → 无限循环。

**修复**: 质量门停滞检测器。连续 `MAX_ENFORCE_DEADLOCK_BLOCKS`(10) 次 enforce 失败后，force-skip 质量门，让终端正常完成。使用已有的 `enforce_deadlock_counters` HashMap（前缀 `qg_stagnation:`）追踪。
