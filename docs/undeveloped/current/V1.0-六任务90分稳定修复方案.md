# V1.0 六任务 90 分稳定修复方案

> 范围：仅 Task 1-6，不纳入 Task 7。
> 原则：修 GitCortex 本体，不手工修任务产物；不修改评分标准，不降低验收口径。
> 目标：所有最终质量问题必须由 GitCortex 自动派发修复终端解决，最终状态只能是 `completed` 或基础设施级 `failed`，不得以 `blocked` 作为用户最终展示状态。

## 3. 系统根因

1. **质量门是判分器，不是执行闭环。** 现有 branch/repo/artifact gate 能发现 blocker，但失败后只是发布错误并停止推进。它没有把 blocker 转换为新的可执行任务，也不会自动启动终端修复。

2. **ExistingCodebase 的“真实接入”要求没有机器化。** Prompt 已经更谨慎，但没有 provider 验证测试是否绑定真实入口、是否遵守包边界、是否复用项目既有状态/i18n/composable/engine 机制，所以 Hoppscotch 和 Kutt 可以完成大量代码，却仍然错接真实应用。

3. **Acceptance Review 粒度不够像评分 rubric。** 当前 review 可以识别明显缺项，但对“运行时风险、重复实现、测试真实性、项目惯用法、安全误报模式”没有一票否决规则，导致中低质量实现被标记 completed。

4. **状态收敛没有 final gate 驱动。** 当所有 terminal/task 已结束但 final gate 仍未通过时，工作流不能停留在 `running` 或展示 `blocked`。它应该立即进入 `running / repairing final issues`，若无法修复再明确 `failed`。

## 4. 修复设计

### 4.1 Final Integration Repair Loop

在 final readiness、branch gate、repo gate、artifact readiness 任一失败时，GitCortex 不再只返回 false，而是创建或复用一个 `Final Integration Repair` task。

Repair task 输入必须包含：
- 失败 gate 类型与 quality run id
- blocker 列表、严重度、相关文件
- 原始用户目标与 workflow strategy
- 已完成 task 摘要和当前分支状态
- 明确要求：真实构建、真实测试、入口 smoke、跨任务集成修复

修复终端提交后自动重新运行 terminal gate、branch gate、repo gate、artifact readiness。全部通过才允许 workflow `completed`。

保护阈值：
- 同一 blocker 连续两轮无文件/commit 变化时，升级 prompt，要求更换修复策略。
- 超过最大 repair rounds 或无法启动 LLM/CLI/终端、仓库不可写时，workflow 标记 `failed`，并保留完整诊断。
- `blocked` 只作为内部诊断，不作为最终用户展示状态。

### 4.2 ExistingCodebase Execution Contract

ExistingCodebase 终端在写代码前必须先识别：
- 真实入口：路由、包导出、server/app 初始化、UI 注册点
- 包边界：monorepo package、workspace、build target
- 项目惯用法：store/composable/i18n/testing convention
- 现有实现：不得重复 engine、重复 store、mock app 替代真实 app

该契约应进入 `build_task_instruction` 和 acceptance review rubric。终端若不能证明真实入口绑定，应继续修复而不是完成。

### 4.3 Rubric Acceptance Review

Acceptance review 从“需求是否满足”升级为 rubric review。以下任一高置信失败必须 `REJECTED`：
- 需求覆盖缺失
- 未接入真实入口或包导出
- 测试只覆盖 fake/mock app 或错误 package
- 违背项目惯用法
- 重复实现已有 engine/store/component/state 机制
- 明显运行时错误
- 高风险安全问题
- 交付不可复现或关键文件未提交

LLM 不可用、changed content 为空、verdict 缺失仍必须 fail-closed。

### 4.4 新质量门 Provider

新增三个指标，并接入 terminal/branch/repo gate：

| 指标 | 阻断示例 |
|------|----------|
| `test_authenticity_issues` | 测试只 import mock Express app；测试覆盖未使用 package；未 import 生产入口 |
| `project_convention_issues` | ESM 包内使用 `require()`；i18n namespace 不匹配；跨包重复 engine/store/types；未复用既有组件/状态机制 |
| `runtime_security_smells` | SQLx `QueryBuilder.push()` 拼接用户输入；Redis `KEYS`；明显未定义变量；危险输入黑名单导致业务误报 |

这些 provider 不能替代评分标准，只负责把已知 58/72/82 分失败模式提前拦截并交给 repair loop。

### 4.5 状态一致性

所有 workflow 状态变化必须同时：
- 写数据库
- 发布 workflow topic
- 广播 WebSocket
- 让工作区详情页和看板失效缓存

服务启动、workflow detail 查询、orchestrator 恢复时执行 reconciliation：
- 若 workflow `running` 但所有 terminal/task 已结束，则立即触发 final gates。
- 若 gates 有 blocker，则进入 repair loop。
- 若 gates 全过，则转 `completed`。
- 若 repair 无法启动，则转 `failed` 并写明原因。

## 7. 验收标准

- 文档中 Task 1-6 每个最终分数都有明细，最终总览与任务明细一致。
- final gate blocker 不再停留为最终 `blocked/running` 展示；必须自动派发修复或明确基础设施级失败。
- repair loop 至少覆盖 branch gate fail、repo gate fail、artifact readiness fail、repair terminal 启动失败四类路径。
- 新 provider 能稳定拦截 Task 2/3/6 的已知扣分模式。
- UI 看板和工作区对话页最终状态一致。
- 最终六任务全量评分全部 ≥90。
