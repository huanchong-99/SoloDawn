# V1.0 GitCortex 统一架构修复方案：消除六任务跷跷板效应

> **创建时间**: 2026-04-24  
> **输入依据**: `docs/undeveloped/current/V1.0-双脑独立架构诊断.md` 全文  
> **目标**: 让 Task 1-6 同时稳定达到 90+，并消除“提高 A 导致 B 下降”的结构性冲突  
> **状态**: 待实施

---

## 一、结论先行

当前分数持续下降，不是因为修复数量不够，而是因为修复落在同一条全局编排路径上。系统缺少一个一等架构对象来表达“这次工作流到底是什么类型、应该用什么质量范围、历史债务如何处理、什么时候才算完成”。

本方案的核心不是继续追加 prompt 或单点 guard，而是引入五个稳定边界：

1. `WorkflowStrategy`: 工作流类型和策略持久化。
2. `QualityScope`: 质量检查范围从裸 `changed_files` 升级为可表达债务边界的 scope。
3. `ReviewGate`: acceptance review 变成完成前硬门禁。
4. `QualityContract`: prompt 质量要求按策略渲染，不再全局一刀切。
5. `FinalReadinessGate`: Branch/Repo 层最终验收，覆盖集成、交付物和评分维度。

只要这五个边界未建立，后续任何修复都会继续在 from-zero 和 existing-codebase 之间来回摆动。

## 二、必须满足的成功标准

### 2.1 分数标准

| 任务 | 类型 | 目标 |
|------|------|------|
| Task 1 知识库 | from-zero | >= 90 |
| Task 2 Hoppscotch | existing-codebase | >= 90 |
| Task 3 Express -> Rust | existing-codebase | >= 90 |
| Task 4 refactor + test | existing-codebase | >= 90 |
| Task 5 微服务电商 | from-zero | >= 90 |
| Task 6 security + monitoring | existing-codebase | >= 90 |

### 2.2 架构标准

1. from-zero 的完整交付要求不得因 existing-codebase 降噪而降低。
2. existing-codebase 的终端不得被未触碰历史债务阻塞。
3. 任何 `passed=false` 的质量门结果必须有可操作 issue、环境阻塞或 pending 状态，不得自动 completed。
4. acceptance review 拒绝必须发生在 task completed 之前。
5. prompt 中的关键交付要求必须有 deterministic verifier。

### 2.3 回归标准

每个阶段合并前必须运行哨兵任务；这不是可选验证。哨兵测试的目的不是替代六任务全量验收，而是在每个 Slice 后快速发现跷跷板是否重新出现。

| 阶段 | 哨兵任务 | 原因 |
|------|----------|------|
| P0 | Task 4 + Task 5 | 一个 existing-codebase，一个 from-zero，最快暴露跷跷板 |
| P1 | Task 6 + Task 5 | 覆盖 security/monitoring 与 greenfield artifact |
| P2 | 全部 6 个任务 | 验证最终评分同时稳定 |

执行顺序上优先跑最短哨兵 Task 4；若 Task 4 出现 existing-codebase 退化，立即停止该 Slice 的推进，不再浪费 3-9 小时跑完整六任务。Task 4 通过后再跑对应的 from-zero 哨兵 Task 5。

## 三、目标架构

### 3.1 新执行链路

```text
Workflow start
  -> detect WorkflowStrategy
  -> persist strategy in OrchestratorState/PersistedState
  -> plan tasks with strategy-aware prompt
  -> dispatch terminal with QualityContract
  -> terminal reports completion
  -> Terminal QualityGate with QualityScope
  -> Acceptance ReviewGate
  -> finalize terminal and mark task completed
  -> Branch gate when task/branch is complete
  -> Repo gate before workflow complete or merge
  -> final readiness decision
```

关键变化：质量、审查、完成、合并不再挤在 `handle_terminal_completed` 的单次事件里做完，而是变成可观察、可测试、可恢复的阶段。

### 3.2 策略对象

建议新增：

```rust
#[derive(Clone, Debug, Serialize, Deserialize, PartialEq, Eq)]
pub enum WorkflowArchetype {
    Greenfield,
    ExistingCodebase,
    HybridFoundation,
}

#[derive(Clone, Debug, Serialize, Deserialize, PartialEq, Eq)]
pub enum QualityProfile {
    FullDelivery,
    IncrementalChange,
}

#[derive(Clone, Debug, Serialize, Deserialize, PartialEq, Eq)]
pub enum DebtPolicy {
    BlockAll,
    BlockIntroducedOnly,
}

#[derive(Clone, Debug, Serialize, Deserialize, PartialEq, Eq)]
pub struct WorkflowStrategy {
    pub archetype: WorkflowArchetype,
    pub quality_profile: QualityProfile,
    pub debt_policy: DebtPolicy,
}
```

推荐映射：

| Archetype | QualityProfile | DebtPolicy | 用途 |
|-----------|----------------|------------|------|
| `Greenfield` | `FullDelivery` | `BlockAll` | Task 1/5 |
| `ExistingCodebase` | `IncrementalChange` | `BlockIntroducedOnly` | Task 2/3/4/6 |
| `HybridFoundation` | `FullDelivery` for foundation, `IncrementalChange` for feature | mixed | 既有仓库内新建独立子系统 |

## 四、P0 修复：先切断根耦合

P0 的目标是让两类任务从同一条管道中分流。没有 P0，后续所有修复仍会互相影响。

### P0-1：持久化 `WorkflowStrategy`

**涉及文件**

| 文件 | 位置 | 修改 |
|------|------|------|
| `crates/services/src/services/orchestrator/state.rs` | `OrchestratorState`，约 `:39`-`:85` | 新增 `workflow_strategy` 字段 |
| `crates/services/src/services/orchestrator/persistence.rs` | `PersistedState`，约 `:23`-`:40` | 新增可序列化字段 |
| `crates/services/src/services/orchestrator/persistence.rs` | state 转换，约 `:113`、`:133`、`:219` | 保存/恢复 strategy |
| `crates/services/src/services/orchestrator/agent.rs` | 初始化/恢复路径，约 `:324`、`:388`、`:497` | 在规划前确定 strategy |

**实施要求**

1. 新增 `WorkflowStrategy::detect(workflow, working_dir)`。
2. 检查仓库已有源码、manifest、目标目录、初始需求关键词。
3. LLM 只能作为 advisory signal，不能作为唯一分类来源。
4. strategy 一旦进入 running 状态后不可静默改变；需要改变必须记录 reason。

**建议检测规则**

```text
if tracked source files == 0 and no package manifest:
    Greenfield
else if task explicitly creates a new standalone app under empty subdir:
    HybridFoundation
else:
    ExistingCodebase
```

源码和 manifest 至少包含：

```text
*.rs, *.ts, *.tsx, *.js, *.jsx, *.py, *.go, *.java, *.kt, *.rb, *.php, *.cs,
Cargo.toml, package.json, pnpm-lock.yaml, pyproject.toml, go.mod, pom.xml,
docker-compose.yml, Dockerfile
```

**验收测试**

1. 空仓库 + from-zero goal -> `Greenfield`。
2. 有 `package.json` 或源码文件 -> `ExistingCodebase`。
3. 既有仓库内要求新增独立服务目录 -> `HybridFoundation`。
4. persistence roundtrip 后 strategy 不丢失。

### P0-2：修复 `order_index > 0` 误判

**现状证据**

`auto_dispatch_initial_tasks` 只在 `task.order_index > 0` 时传递 foundation context 和 initial goal（`crates/services/src/services/orchestrator/agent.rs:5779`-`:5906`）。`dispatch_next_terminal` 也在后续终端调度中依赖同类逻辑（`crates/services/src/services/orchestrator/agent.rs:3042`-`:3126`）。

这对 Greenfield foundation 合理，但对 ExistingCodebase 的第一个任务是错误的：existing-codebase 的首任务通常也是 `order_index == 0`，却不是 foundation。

**修改点**

在 `auto_dispatch_initial_tasks` 和 `dispatch_next_terminal` 中统一改成：

```rust
let should_pass_initial_goal = match strategy.archetype {
    WorkflowArchetype::Greenfield => task.order_index > 0,
    WorkflowArchetype::HybridFoundation => task.order_index > 0 || task_is_existing_code_task(&task),
    WorkflowArchetype::ExistingCodebase => true,
};

let should_pass_foundation_context = match strategy.archetype {
    WorkflowArchetype::Greenfield | WorkflowArchetype::HybridFoundation => task.order_index > 0,
    WorkflowArchetype::ExistingCodebase => false,
};
```

**验收测试**

1. ExistingCodebase 的 `order_index == 0` 任务必须收到 `initial_goal`。
2. Greenfield 的 foundation 任务不得收到大量 feature-level foundation context。
3. Greenfield 的 feature 任务必须收到 foundation context。
4. 指令中不得出现与策略矛盾的要求。

### P0-3：completion/review 状态分离

**现状证据**

`handle_terminal_completed` 同时处理质量门、终端终结、任务状态、acceptance review 和后续调度（`crates/services/src/services/orchestrator/agent.rs:1330`-`:1744`）。当前流程中 `try_finalize_terminal` 和 shutdown 发生在 acceptance review 之前（约 `:1509`-`:1517`），review 在之后运行（约 `:1619`-`:1671`）。

**修改点**

把 `handle_terminal_completed` 拆成阶段函数：

```text
handle_terminal_completed
  -> maybe_run_terminal_quality_gate
  -> run_acceptance_review_gate
  -> finalize_terminal_after_review
  -> update_task_completion
  -> dispatch_followups_or_final_gates
```

新增或激活状态：

```text
terminal_completed_reported
quality_pending
review_pending
review_rejected
completed
```

`TASK_STATUS_REVIEW_PENDING` 已存在于 `constants.rs` 附近，但应统一为可读值 `review_pending` 并真正接入状态机。

**强制规则**

1. `run_acceptance_review` 通过之前不得调用 `try_finalize_terminal`。
2. review rejected 时任务不得 marked completed。
3. LLM unavailable 不得 auto approve；应进入 `review_pending` 或可重试状态。
4. changed file content 为空不得 auto approve；除非该任务声明为 no-code task 且有明确 waiver。
5. review 必须锁定 quiet window 结束时的 commit hash；review prompt、changed files、diff stat 都基于同一个 locked commit。
6. 如果 review 期间终端又产生新 commit，当前 review 结果必须失效，流程重新进入 `Terminal QualityGate -> quiet window -> ReviewGate`。
7. `pending_quiet_completion_checks` 只表示“等待安静窗口结束”，不能同时表示“review 已经针对最终代码完成”。

**验收测试**

1. 构造 review rejected，断言 terminal/task 不进入 completed。
2. 构造 LLM unavailable，断言进入 review_pending。
3. 构造 empty changed content，断言不自动 approved。
4. rejected 后 fix instruction 能送达到仍可用的终端或新 relaunch terminal。
5. review 期间出现新 commit，断言旧 review 被丢弃并重新触发 quality/review。

## 五、P1 修复：让质量门既严格又不误伤

P1 的目标是把“严格度”从全局开关改成范围感知。from-zero 仍全量严格，existing-codebase 只阻塞新增问题。

### P1-1：引入 `QualityScope`

**现状证据**

`handle_checkpoint_quality_gate` 当前以 `hash~1` 收集单提交变更（`crates/services/src/services/orchestrator/agent.rs:2049`-`:2052`），质量引擎入口仍接收 `changed_files: Option<&[String]>`（`crates/quality/src/engine.rs:111`-`:116`）。Provider trait 也只知道裸 `changed_files`（`crates/quality/src/provider/mod.rs:140`-`:148`）。

**新增模型**

```rust
pub enum QualityScope {
    IntroducedOnly {
        merge_base: String,
        head: String,
        files: Vec<String>,
    },
    TaskBranch {
        merge_base: String,
        head: String,
    },
    RepoFinal {
        head: String,
    },
}
```

**映射规则**

| WorkflowStrategy | Terminal gate | Branch gate | Repo gate |
|------------------|---------------|-------------|-----------|
| Greenfield | `TaskBranch` 或 `RepoFinal` | `TaskBranch` | `RepoFinal` |
| ExistingCodebase | `IntroducedOnly` | `IntroducedOnly` + integration checks | limited `RepoFinal` only for requested areas |
| HybridFoundation | foundation 用 full，feature 用 introduced | mixed | profile-driven |

**修改点**

1. Slice 4 第一版先实现兼容层，不直接大改所有 provider 签名。
2. `QualityScope` 先存在于 orchestrator/engine 边界，向旧接口折叠：`IntroducedOnly -> Some(files)`，`TaskBranch/RepoFinal -> None`。
3. `QualityEngine::run_with_scope` 可作为新入口，内部复用旧 `run(level, changed_files)`，避免一次性修改 9 个 provider。
4. `handle_checkpoint_quality_gate` 用 merge-base 计算 branch scope，不再只用 `hash~1`。
5. Provider 全量接收 `QualityScope` 是第二阶段改造；必须等分流效果验证后再推进。
6. 对不能安全增量化的 provider，第一版只允许在 `RepoFinal` 或明确 opt-in 的 Branch gate 中运行。

**验收测试**

1. existing-codebase 中预存 lint/security 问题不应阻塞未触碰任务。
2. existing-codebase 中新引入 secret/weak default 必须阻塞。
3. greenfield 中缺少 tests/README/env/Docker 仍必须阻塞。
4. 兼容层下旧 provider 测试不需要一次性重写；`IntroducedOnly` 能正确传递 changed files 即可。

### P1-2：Provider 适用性修复

**现状证据**

`QualityProvider::applicable_metrics` 默认返回所有 supported metrics（`crates/quality/src/provider/mod.rs:158`-`:164`）。`RustProvider` 未覆盖该方法，并且在没有 `Cargo.toml` 或工具失败时写入 `MeasureValue::Int(-1)`（`crates/quality/src/provider/rust_analyzer.rs:217`-`:320`）。Evaluator 将 `-1` 视为 ERROR（`crates/quality/src/gate/evaluator.rs:50`-`:63`）。

**修改点**

1. `RustProvider::applicable_metrics` 在无 Rust target 时返回空。
2. `FrontendProvider` 保留现有 target-aware 行为，但统一使用 `QualityScope`。
3. `CompletenessProvider` 在 `ExistingCodebase + IntroducedOnly` 下只检查本次新增/修改模块的测试承诺，不扫描全仓。
4. `SecurityProvider` 对 existing-codebase 只阻塞新增依赖漏洞、新增 secret、新增弱默认；全仓 audit 放到 Repo gate 且受 strategy 控制。
5. `BuiltinCommonProvider` 扩展输入文件类型，覆盖 yaml/env/json/toml/docker/markdown artifacts，但按规则类型过滤。

**验收测试**

1. 非 Rust 项目不产生 Rust metric ERROR。
2. 无工具或工具不可用时生成明确 infra issue，而不是裸 `-1`。
3. Provider 对不适用 scope 返回空 metrics，不造成 false failure。

### P1-3：禁止质量门 fail-open

**现状证据**

`QualityReport::aggregate` 只统计 provider issues（`crates/quality/src/report.rs:35`-`:45`）。因此 evaluator ERROR 可能导致 `passed=false`，但 summary 中 `total_issues=0`、`blocking_issues=0`。随后 `handle_quality_gate_result` 会把它当作指标采集失败并推进 completed（`crates/services/src/services/orchestrator/agent.rs:2346` 起，关键分支约 `:2369`-`:2374`）。

**修改点**

1. 在 `crates/quality/src/engine.rs` 中，`ConditionEvaluator::evaluate_all` 后将每个 ERROR 转成 synthetic `QualityIssue`。
2. issue id 规则：`quality_gate::<metric_key>::evaluation_error`。
3. severity 固定为 blocker，category 为 `infra` 或 `quality_gate`。
4. `handle_quality_gate_result` 删除 `passed=false,total=0,blocking=0` 自动完成分支。
5. 环境阻塞必须显式标记，例如 `quality_gate::tool_unavailable`。

**验收测试**

1. 构造 missing metric，必须产生 blocking issue。
2. 构造 `MeasureValue::Int(-1)`，不得自动 completed。
3. 环境阻塞必须进入 pending/blocking，不污染 task completed。

## 六、P1 修复：策略化 prompt 合约

### P1-4：用 `QualityContract` 替代全局 suffix

**现状证据**

`dispatch_terminal` 当前无条件追加 `QUALITY_REQUIREMENTS_SUFFIX`（`crates/services/src/services/orchestrator/agent.rs:5121`-`:5135`）。该 suffix 位于 `constants.rs`（`crates/services/src/services/orchestrator/constants.rs:95` 起），同时要求 README、Docker、compose、CI、测试、安全和格式化。

**目标**

全局 suffix 拆成策略渲染：

```rust
pub struct QualityContract;

impl QualityContract {
    pub fn render(strategy: &WorkflowStrategy, task_phase: TaskPhase, terminal_role: TerminalRole) -> String;
}
```

**Greenfield contract**

必须保留：

1. 可运行的项目骨架。
2. README/setup/run/test 文档。
3. `.env.example`。
4. Dockerfile / compose，除非技术栈明确不需要。
5. CI 或等价验证脚本。
6. 测试覆盖关键业务路径。
7. secret/weak-default 检查。

**Existing-codebase contract**

必须改为：

1. 遵循现有架构和脚本。
2. 只修改任务相关文件。
3. 只修复本次引入的问题。
4. 为新增/修改逻辑补测试。
5. 不新增 Docker/CI/README，除非任务明确要求。
6. 如果现有构建失败，区分 pre-existing 与 introduced。

**指令预算**

| 类型 | 目标上限 |
|------|----------|
| Greenfield foundation instruction | <= 12k chars |
| Greenfield feature instruction | <= 10k chars |
| Existing-codebase instruction | <= 8k chars |
| QualityContract suffix | <= 2k chars |

**验收测试**

1. Greenfield prompt 包含 artifact 要求。
2. Existing-codebase prompt 不包含“创建 Docker/CI/README”的默认要求。
3. Existing-codebase `order_index == 0` prompt 包含 initial goal。
4. prompt 长度超过预算时测试失败或记录 warning。

### P1-5：策略化 acceptance review prompt

**现状证据**

`run_acceptance_review` 在 `agent.rs` 约 `:3941` 起；当前空内容和 LLM 不可用会 auto approve（约 `:3952`、`:3988`）。`AcceptanceReviewResult::parse` 在 verdict 缺失时默认 approved（`crates/services/src/services/orchestrator/types.rs:473`-`:520`）。

**修改点**

1. Greenfield review 检查完整交付物矩阵。
2. Existing-codebase review 检查目标变更、兼容性、测试和无范围蔓延。
3. verdict 缺失必须视为 invalid/pending，不得 approved。
4. changed content 必须覆盖 artifacts：`README*`、`.env.example`、`.gitignore`、`.github/workflows/*`、`Dockerfile*`、`docker-compose*.yml`、`*.md`、`*.yaml`、`*.yml`、`*.toml`。

**验收测试**

1. REVIEW JSON 无 verdict -> rejected 或 pending。
2. LLM unavailable -> review_pending。
3. Greenfield 缺 README/env/tests -> rejected。
4. Existing-codebase 出现无关 Docker/CI 变更 -> rejected，除非任务要求。

## 七、P2 修复：最终交付门

### P2-1：执行 Branch/Repo gate

**现状证据**

配置文件实际路径为 `quality/quality-gate.yaml`，其中 `mode: enforce` 在 `:12`，`terminal_gate` 在 `:17`，`branch_gate` 在 `:63`，`repo_gate` 在 `:114`。Orchestrator 当前实际运行的是 `QualityGateLevel::Terminal`（`crates/services/src/services/orchestrator/agent.rs:2092`-`:2097`）。

**修改点**

1. P2-1 必须依赖 P0-3 的 `handle_terminal_completed` 拆分完成后再实施。
2. Branch gate 不应硬塞回未拆分的 `handle_terminal_completed`，而应插入拆分后的 `dispatch_followups_or_final_gates` 阶段。
3. 任务分支完成后运行 Branch gate。
4. workflow complete 或 merge 前运行 Repo gate。
5. `auto_sync_workflow_completion`、`execute_auto_merge`、`trigger_merge`、`CompleteWorkflow` 都必须检查 final readiness。
6. Branch/Repo gate 使用 `WorkflowStrategy` 决定是否全仓阻塞。

**建议流程**

```text
Terminal gate pass
  -> ReviewGate pass
  -> task completed
  -> Branch gate
  -> all tasks completed?
  -> Repo gate
  -> artifact readiness
  -> workflow complete / merge
```

**验收测试**

1. Terminal pass 但 Branch fail，不得 workflow complete。
2. Branch pass 但 Repo fail，不得 merge。
3. Existing-codebase Repo gate 不得因无关历史债务失败。
4. Greenfield Repo gate 必须全量检查交付物。

### P2-2：Artifact readiness matrix

**目标**

把 prompt 中的交付要求变成确定性 verifier。

第一版 Artifact Readiness 只做低成本确定性检查：文件存在、非空、路径合理、与任务策略匹配。内容语义检查先交给 acceptance review，避免在 provider 中重复建设跨文件语义分析。

| Artifact | Greenfield | Existing-codebase | V1 Verifier | V2 Verifier |
|----------|------------|-------------------|-------------|-------------|
| README/setup | required | only if touched/requested | file exists + non-empty | contains run/test/setup |
| `.env.example` | required if env vars used | only if new env introduced | file exists + non-empty when env changed | env keys match code references |
| Dockerfile | required for service app | only if requested | file exists + non-empty | buildable syntax / referenced scripts |
| compose | required for multi-service | only if requested | file exists + non-empty | service/env/port coherence |
| CI | required or verification script alternative | only if requested | workflow or verification script exists | runs build/test commands |
| tests | required | required for modified logic | test file exists or changed | test target maps to changed logic |
| security defaults | required | introduced only | existing secret/weak-default scanners | no hardcoded secrets / weak defaults |

**实现位置**

建议新增 provider：

```text
crates/quality/src/provider/artifact_readiness.rs
```

并接入：

```text
crates/quality/src/provider/mod.rs
crates/quality/src/engine.rs
quality/quality-gate.yaml
```

**验收测试**

1. Greenfield 缺 artifact 被明确阻塞。
2. Existing-codebase 不因缺 Docker/CI 被阻塞。
3. Existing-codebase 新增 env var 但无 `.env.example` 更新会失败。
4. V1 不做 env key semantic matching；该检查必须由 review prompt 覆盖并在 V2 再落入 provider。

## 八、实施顺序与提交切片

### Slice 1：Strategy 状态与检测

包含：

1. `WorkflowStrategy` 类型。
2. `OrchestratorState` / `PersistedState` 字段。
3. 检测函数和持久化 roundtrip 测试。

不包含：

1. prompt 大改。
2. quality provider 大改。

验收：

```text
cargo test -p services orchestrator::state
cargo test -p services orchestrator::persistence
```

### Slice 2：Dispatch 分流

包含：

1. `auto_dispatch_initial_tasks` 使用 strategy。
2. `dispatch_next_terminal` 使用 strategy。
3. `build_task_instruction` 接收 strategy/phase。
4. prompt budget 测试。

验收：

```text
cargo test -p services build_task_instruction
cargo test -p services auto_dispatch_initial_tasks
```

### Slice 3：ReviewGate 前移

包含：

1. 拆分 `handle_terminal_completed`。
2. review pending 状态。
3. fail-open 移除。
4. artifact content collection 扩展。

验收：

```text
cargo test -p services acceptance_review
cargo test -p services terminal_completed
```

### Slice 4：QualityScope 与 Provider 适用性

包含：

1. `QualityScope` 类型。
2. `run_with_scope` 兼容入口和 scope -> changed_files 适配。
3. Rust provider applicability 修复。
4. completeness/security/builtin_common 的最小增量化策略。
5. synthetic issue 生成。

不包含：

1. 一次性修改全部 provider trait 签名。
2. 强制所有 provider 第一版都理解完整 `QualityScope`。

验收：

```text

```

### Slice 5：QualityContract 与 artifact readiness

包含：

1. 替换全局 suffix。

4. Branch/Repo gate 接入。

验收：

```text

```

### Slice 6：六任务回归

包含：

1. 重置测试环境。
2. 按固定种子/固定任务集执行六任务。
3. 记录每任务分数、失败原因、是否触发 baseline。
4. 任一任务低于 90 或任一 from-zero/existing-codebase 出现互相拉低，回到对应 slice 修复。

执行约束：

1. Slice 6 是最终验收，不是每个小修改后的默认验证。
2. 每个 Slice 合并前必须先跑第 2.3 节定义的哨兵任务。
3. 如果哨兵失败，不进入六任务全量测试。
4. 六任务全量测试建议并行化执行；若只能顺序执行，先跑 Task 4、Task 5、Task 6，再跑其余任务。

## 九、关键风险与控制

### 风险 1：Strategy 检测误判

控制：

1. 检测结果写入日志和 workflow state。
2. 允许显式 override，但必须记录 reason。
3. HybridFoundation 作为缓冲类型，避免二元分类过硬。

### 风险 2：Existing-codebase 放过真正问题

控制：

1. `IntroducedOnly` 只豁免历史问题，不豁免新增问题。
2. 新增依赖、新增 secret、新增 env、新增公开 API 必须被检查。
3. Branch gate 检查集成层，防止局部变更破坏整体。

### 风险 3：Greenfield 被过度分阶段拖慢

控制：

1. Foundation 只负责骨架和共享基线，不吞掉功能实现。
2. Feature task 必须收到 foundation context 和 initial goal。
3. Repo gate 才做最终全量 artifact readiness。

### 风险 4：ReviewGate 前移造成卡死

控制：

1. review_pending 有最大重试和清晰状态。
2. LLM 不可用归类为 infra blocker。
3. 允许人工 override，但需要记录 waiver。

### 风险 5：Provider 改造范围过大

控制：

1. 先实现 trait 兼容层，旧 provider 可临时把 `QualityScope` 映射回 changed files。
2. 按 provider 分批改造。
3. 所有 fail-open 分支先关闭，再逐步优化误报。

## 十、与既有 10 项修复的兼容性

本方案不是回滚既有 10 项修复，而是在其上增加稳定边界。实施时禁止因为重构而删除既有有效修复。

| 既有修复 | 处理方式 | 强制要求 |
|----------|----------|----------|
| 1. LLM acceptance review | 保留并前移为 `ReviewGate` | 不得退回完成后软审查 |
| 2. Foundation-only completion bug fix | 被 `WorkflowStrategy` 取代为更强机制 | 保留防御性 guard，直到新 strategy 覆盖测试稳定 |
| 3. General quality checks | 保留但 scope-aware | 不得全局关闭，只能按 strategy 限定范围 |
| 4. Completion guards | 保留并迁移到状态机 | 不得因拆分 `handle_terminal_completed` 删除保护 |
| 5. Full-branch diff | 保留并扩展到 quality gate | merge-base 逻辑不能退回 `commit~1` |
| 6. 128KB review limit | 明确保留 | `ACCEPTANCE_REVIEW_MAX_BYTES` 不得降回 32KB |
| 7. 4000-char task descriptions | 保留并纳入 instruction budget | 不得重新压回 200 字符 |
| 8. Foundation context passing | 保留但只给需要 foundation 的任务 | ExistingCodebase 默认不注入 foundation 文件树 |
| 9. initial_goal propagation | 保留并修复 `order_index == 0` 漏传 | ExistingCodebase 所有任务必须收到 initial goal |
| 10. LLM input generation for `PromptKind::Input` | 明确保留 | Prompt handler 重构不得移除自动输入生成 |

尤其注意两点：第一，128KB review limit 是大型任务验收不被截断的基础；第二，`PromptKind::Input` 的 LLM 输入生成属于自治能力修复，不属于本次跷跷板架构边界，但必须继续保留。

## 十一、副脑审计反馈后的强制修订

本节记录副脑独立审计后已经采纳的方案修订，防止实施时回到旧版本。

1. **QualityScope 分阶段落地**: Slice 4 第一版必须使用兼容层，不能一次性强改 9 个 provider 签名。
2. **ReviewGate 锁定 commit**: review 只审 quiet window 结束时的 locked commit；review 期间新 commit 会使本轮 review 失效。
3. **Branch/Repo gate 依赖 P0-3**: P2-1 必须插入拆分后的 `dispatch_followups_or_final_gates`，不得硬塞回旧的巨型 `handle_terminal_completed`。
4. **Artifact readiness V1 降复杂度**: 第一版只做存在、非空、策略匹配；跨文件语义交给 acceptance review 和 V2 provider。
5. **哨兵测试强制化**: 每个 Slice 合并前必须先跑哨兵；哨兵失败不得进入六任务全量测试。
6. **既有修复保留**: 128KB review limit 和 `PromptKind::Input` 生成必须显式保留。

## 十二、完成定义

本修复方案完成必须同时满足：

1. 代码层存在持久化 `WorkflowStrategy`，并覆盖 state/persistence roundtrip。
2. `order_index > 0` 不再承担“是否 foundation”的架构语义。
3. `QUALITY_REQUIREMENTS_SUFFIX` 不再无条件追加到所有终端。
4. acceptance review 在 terminal/task completed 之前执行。
5. `run_acceptance_review`、`AcceptanceReviewResult::parse` 不再 fail-open。
6. Quality gate 不再出现 `passed=false,total_issues=0,blocking_issues=0` 自动 completed。
7. Provider 对 scope/applicability 有明确声明。
8. `quality/quality-gate.yaml` 的 Branch/Repo gate 被 orchestrator 实际调用。
9. Greenfield artifact 要求有 verifier。
10. 六任务全量测试全部 >= 90，且没有 from-zero 与 existing-codebase 互相拉低。

---

## 十三、后续实施建议

第一步不要先改质量 provider，也不要先继续加强 prompt。优先做 `WorkflowStrategy` 和 completion/review 状态拆分，因为这是根耦合。如果先改 provider 或 prompt，仍然会落回同一个全局管道。

推荐立即创建实施任务：

1. `P0-Strategy-State`: strategy 类型、检测、持久化。
2. `P0-Strategy-Dispatch`: initial_goal/foundation_context 分流。
3. `P0-ReviewGate`: review before finalize，移除 fail-open。
4. `P1-QualityScope`: scope + provider applicability + synthetic issues。
5. `P1-QualityContract`: 策略化 prompt 与 artifact verifier。
6. `P2-FinalReadiness`: Branch/Repo gate 与六任务全量验收。

这套顺序的关键点是：先建立边界，再提高严格度。边界稳定后，from-zero 的完整性和 existing-codebase 的最小侵入性会成为两条互不牵制的路径，而不是同一个旋钮的两端。
