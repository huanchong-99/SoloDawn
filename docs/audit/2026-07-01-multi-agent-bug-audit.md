# SoloDawn — Multi-Agent Bug Audit (2026-07-01)

Branch: `claude/exciting-ritchie-afa53g-bugfix`

## Method

A fan-out audit orchestrated as a workflow: **11** documentation-research agents (official docs for every embedded component), **6** architecture agents, **26** per-subsystem bug-finder agents, then an **adversarial verify** stage (refute-first) and an **adversarial fix-review** stage gating every fix before it was eligible for submission. First run reached the session token limit after find+architecture; it was **resumed from its run journal** so the verify+review phases completed. 152 agent-runs total.

**Findings:** 102 raw (3 critical, 25 high, 32 medium, 42 low). After adversarial verification + review, fixes were applied in two waves.

### Environment constraint

The `ts-rs` dependency is a **git dependency on github.com**, which this environment's egress policy blocks (403). The Rust workspace therefore **could not be compiled here**; all Rust fixes were reviewed by hand + passed the adversarial review gate, and each was individually checked for symbol/type/borrow safety against the surrounding code. The **frontend was fully verified** (`tsc` + ESLint + vitest all green).

## Architecture & long-term maintainability assessment

### AI orchestrator subsystem (crates/services/src/services/orchestrator/*) plus the executor connector layer (cra

The subsystem has two very different maintainability profiles. The peripheral abstractions are genuinely well-designed: the LLM layer (llm.rs, resilient_llm.rs) is a clean trait + decorator stack (LLMClient trait -> RateLimitedClient -> ResilientLLMClient) with a format-driven factory, so swapping/adding an LLM provider is a bounded, discoverable change. Connectors (coding agents) live in a separate executors crate behind a StandardCodingAgentExecutor enum_dispatch trait with 9 existing implementations, so adding a new agent is a well-trodden path. The data models (OrchestratorInstruction action-DSL, BusMessage, OrchestratorState, WorkflowStrategy, audit rubric) are clear, serde-tagged, and well-documented, and the core event loop (agent.rs run() -> select! on bus + watchdog sweeps -> handle_message exhaustive match) is intelligible. The dominant risk is agent.rs itself: a single 13,890-line file with a 216-method God-object impl on OrchestratorAgent, 280 direct DB-model references and no repository seam, meaning most orchestration logic is only integration-testable against a live sqlite DB. Individual methods are readable and heavily traced, but the sheer size and the coupling to db::models make the whole intelligible only in fragments and slow to extend safely.

**Strengths:** Clean LLM abstraction: LLMClient trait with layered decorators (RateLimitedClient, ResilientLLMClient) and a format-driven factory make providers truly swappable (llm.rs, resilient_llm.rs); ResilientLLMClient is a well-modeled circuit-breaker/failover with explicit ProviderEvent enum, exponential backoff, and a 6h retry budget for transient failures (resilient_llm.rs); Coding-agent connectors are a proven extension point: StandardCodingAgentExecutor + enum_dispatch with 9 implementations in a separate crate (executors/src/executors/mod.rs:192); Data models are clear and serde-tagged: OrchestratorInstruction action-DSL, BusMessage, WorkflowStrategy derivation, and the embedded audit rubric are all self-documenting; Core event loop is intelligible: agent.rs run() is a clean select! over bus + watchdog sweeps, and handle_message is an exhaustive match with an explicit no-op arm that forces compile errors on new BusMessage variants (agent.rs:1101, 1124-1139)

**Risks:** agent.rs is a 13,890-line God object: one impl on OrchestratorAgent with 216 methods (155 async). This is the single biggest long-term maintainability liability and the main reason the whole will not be intelligible in 1 month; Deep coupling to concrete DB models: 280 direct db::models/self.db.pool references with no repository seam, so most orchestration logic is integration-testable only and unit-testing decision logic requires a live sqlite DB; Pure decision helpers (parse_instructions, extract_json_from_mixed_response, project-signal classification) appear to lack dedicated unit tests despite being trivially unit-testable and high-value; Message bus uses an enum wrapper (MessageBus::InMemory/Redis) rather than a trait, so a third backend touches every delegating match arm (message_bus.rs:430); Mixed Chinese/English doc comments in types.rs and config.rs raise the onboarding bar for part of the team

**Recommendations:** Split agent.rs into responsibility-scoped impl submodules (terminal_completion, quality_gate, git_events, stall_recovery, instruction_parse, dispatch) - mechanical, no logic change, biggest single win for 1-month intelligibility; Extract and unit-test the pure helpers (parse_instructions, extract_json_from_mixed_response, project-signal/goal classifiers) that guard the fragile LLM-output boundary; Introduce a thin repository trait over the agent's DB access so decision methods become unit-testable against a fake instead of a live sqlite pool; Validate api_type in OrchestratorConfig::validate() instead of silently defaulting unknown values to OpenAIChat in resolve_endpoint; Lift the pinned Anthropic api-version ('2023-06-01', llm.rs:490) into a named, documented, config-overridable constant; Promote MessageBus enum to a BusBackend trait opportunistically so new transports don't touch every match arm

### Executor abstraction (crates/executors/src/executors/*) — long-term maintainability, extensibility, and swappa

The executor layer is a genuinely well-factored plugin system built around one narrow trait (StandardCodingAgentExecutor) fanned out via enum_dispatch over the CodingAgent enum, with strong shared infrastructure (CommandBuilder + CmdOverrides for argv assembly, ExecutionEnv for env/security, AcpAgentHarness for ACP agents, a normalized log/entry data model, and a profile/variant config system). The per-agent files are short, uniform, and readable (gemini/qwen/amp/droid/copilot are each ~130-320 lines and follow an obvious template), so adding or swapping a straightforward CLI agent is a rapid, low-risk operation. The main long-term risks are (a) the trait conflates process-spawning with log-normalization and session semantics, pushing complexity into a few outlier files (claude.rs is 120KB and is reused as the log processor by amp), (b) adding a new agent requires touching ~8 scattered sites, several of which are NOT compiler-enforced (a _ => catch-all in get_mcp_config, matches!/!= gates in container.rs, and a hand-maintained decl() list in generate_types.rs) so a partially-wired agent compiles but silently misbehaves, and (c) test coverage is extremely uneven: claude has 23 tests, everything else (codex, opencode, gemini, qwen, amp, copilot, droid) has essentially zero unit tests, so regressions in the shared spawn/session/log paths are unguarded.

**Strengths:** Single narrow trait (StandardCodingAgentExecutor) with well-chosen defaults (spawn_review, get_availability_info, get_setup_helper_action) dispatched by enum_dispatch — new agents implement a minimal surface; Agent-agnostic normalized log model (NormalizedEntry/ActionType/ToolStatus in logs/mod.rs) fully decouples every agent's wire format from the frontend; this is the architectural keystone; Shared, well-tested infrastructure: CommandBuilder/CmdOverrides centralize argv+override logic (command.rs, 8 unit tests), ExecutionEnv centralizes env injection with a real security policy (env.rs BLOCKED_ENV_VARS, PORT stripping, PATH-append); AcpAgentHarness cleanly reuses one ACP driver across gemini and qwen via a session_namespace parameter, and is ready for more ACP agents with with_model/with_mode builders; Uniform per-agent file template (append_prompt + cmd overrides + build_command_builder + 3 trait methods) makes the simple agents almost boilerplate — high consistency, low cognitive load

**Risks:** Adding an agent touches ~8 sites, several NOT compiler-enforced: get_mcp_config() has a `_ =>` catch-all (silent wrong MCP layout), container.rs uses matches!/!= BaseCodingAgent gates (silent wrong branch), and generate_types.rs holds a hand-maintained decl()/generate_json_schema() list (omit -> TS types silently missing). A partially-wired agent compiles and runs.; claude.rs is a 120KB god-file mixing the executor, the shared stream-json ClaudeLogProcessor (reused by amp via HistoryStrategy::AmpResume), and ~15 wire-format structs — amp-driven changes can silently regress claude; it is the least modular part and the hardest to reason about in a month; Test coverage is severely uneven: codex/opencode/gemini/qwen/amp/copilot/droid executor bodies have ZERO unit tests; only claude (23) and the shared command/env helpers are covered. Regressions in spawn/session/normalize paths are unguarded.; spawn/spawn_follow_up build tokio Command and spawn processes/background tasks directly with no injection seam — argv/env/session logic cannot be asserted without the real agent binary, which is why per-agent tests are absent; Session-id/resume handling is bespoke per agent (copilot regex-scrapes a logfile, amp shells out to `threads fork`, droid fork_session, ACP uses SessionManager) with no shared SessionSource abstraction — most error-prone area to extend and reason about

**Recommendations:** Add a single documented 'new agent checklist' (a doc comment at the top of executors/mod.rs plus a CONTRIBUTING note) enumerating every touchpoint: mod.rs module+variant, capabilities(), get_mcp_config(), preconfigured_mcp()/mcp_config.rs adapter, default_profiles.json, generate_types.rs decl+schema, and any container.rs gate. This is the cheapest, highest-leverage fix for the scattered-registration risk.; Turn silent fan-out sites into compiler-enforced ones: replace the `_ =>` catch-all in CodingAgent::get_mcp_config() (mod.rs ~line 137) with explicit per-variant arms so a new agent forces a decision; audit container.rs matches!/!= BaseCodingAgent gates and give them explicit exhaustive matches or a documented default.; Add a build-time or unit-test assertion that every CodingAgent variant has (a) a default_profiles.json entry whose BaseCodingAgent matches and (b) a decl()/schema registration in generate_types.rs — a single test iterating BaseCodingAgent::VARIANTS would catch omitted wiring at CI time instead of at runtime.; Split claude.rs into a claude/ submodule (executor.rs, log_processor.rs, wire_types.rs) mirroring the existing codex/ and acp/ layout; move ClaudeLogProcessor + HistoryStrategy into a shared logs module so amp's dependency on it is explicit rather than reaching into the claude executor file.; Backfill pure-function unit tests for every agent's build_command_builder() (argv assembly is deterministic and side-effect-free) — this covers the most change-prone per-agent logic (flags, model, autonomy, yolo) with cheap tests and no process spawning.; Introduce a small shared trait/helper for session extraction (e.g. a SessionSource enum: StdoutPrefix, LogFileRegex, ForkCommand) so copilot/amp/droid/acp session handling is expressed as data/config rather than four unrelated bespoke implementations — reduces the risk surface when adding resumable agents.

### HTTP/WS API layer (crates/server/src/routes, middleware, mcp): route organization, ts-rs DTO sharing, error mo

The API layer is a well-organized, modular axum 0.8 codebase with strong bones: a single centralized `ApiError` -> `IntoResponse` error model, per-resource route modules composed via `merge`/`nest` in `routes/mod.rs`, entity-loader middleware that injects DB models as request extensions, and a genuinely swappable executor/agent abstraction (`CodingAgent` enum + `StandardCodingAgentExecutor` trait via enum_dispatch). Data models are shared with the frontend through ts-rs. It will remain intelligible after 1 month for most modules and can be extended in 6 months. The two structural liabilities are (1) a 260-entry MANUAL `.decl()` list in `bin/generate_types.rs` that is the single source of TS type export — a silent drift hazard since most route DTOs are not `#[ts(export)]` and depend on being remembered there; and (2) a handful of files (workflows.rs at 5102 lines, task_attempts.rs at 1930) that concentrate too much, plus repeated boilerplate (the auth-gate block is copy-pasted 6x in model_loaders.rs). Connector extensibility is advertised but not real: chat_integrations claims "provider-agnostic" yet hardcodes telegram. Response-envelope consistency has one leak (models.rs bypasses ApiResponse). None of these block extension; they raise the per-change tax and the risk of subtle omissions.

**Strengths:** Centralized, exhaustive error model: single ApiError enum with `#[from]` conversions and one IntoResponse impl that maps every domain error to a status code + stable error_type string, with deliberate internal-error redaction (G35-004) and unit tests for the mapping (error.rs).; Clean per-resource route modularity: each routes/*.rs exposes a `router()`/`*_routes()` returning `Router<DeploymentImpl>`, composed via merge/nest in routes/mod.rs; adding a resource is a one-line change.; Genuinely swappable agent abstraction: CodingAgent enum + enum_dispatch StandardCodingAgentExecutor trait with defaulted methods and auto-derived discriminants (executors/executors/mod.rs) — 9 agents integrated behind one trait.; Entity-loader middleware pattern (model_loaders.rs) injects Project/Task/Workspace/etc. as request extensions, keeping handlers thin and 404/500 handling uniform.; Security-conscious details are well-documented and correct: CORS layered outside auth so OPTIONS preflight works (W2-34-04), constant-time token comparison, and delete_task closes a TOCTOU by scoping the DELETE with project_id (W2-18-08).

**Risks:** TS type export is a 260-entry MANUAL `.decl()` list in bin/generate_types.rs; most route DTOs are NOT `#[ts(export)]` (e.g. tasks.rs CreateAndStartTaskRequest, ShareTaskResponse) and appear in shared/types.ts only because someone remembered to add them. A new/renamed DTO silently omitted breaks frontend type safety with no compile error — `--check` only fires if a maintainer already added the entry.; Size concentration: workflows.rs is 5102 lines (18 routes + handlers + logic) and task_attempts.rs is 1930; these are the modules most likely to become unintelligible and merge-conflict-prone. Only task_attempts has begun splitting into submodules.; Advertised-but-fake extension seams: chat_integrations.rs documents a 'provider-agnostic' `/chat/{provider}` entrypoint but hardcodes CHAT_EVENT_ALLOWED_PROVIDER='telegram'; models.rs dispatches LLM providers via a hardcoded string match with hardcoded base URLs. Adding a provider means editing core files, not registering an impl.; Response-envelope inconsistency: 287 sites use ApiResponse::success/error, but models.rs returns bare ModelsListResponse/VerifyModelResponse, so clients must special-case those endpoints.; Copy-paste boilerplate: the RequestContext-lookup + assert_authorized block is duplicated verbatim 6x across the loader middlewares (model_loaders.rs) and again inline in handlers like get_tasks — a change to the auth posture must be made in many places.

**Recommendations:** Kill the manual ts-rs drift risk: in bin/generate_types.rs, replace (or backstop) the hand-curated 260-entry list with ts-rs `#[ts(export)]` + `export_all`/directory export, OR add a CI test that reflects over all `#[derive(TS)]` types in the workspace and fails if any is missing from the decl list. This single change removes the most likely source of silent frontend breakage.; Split the two giant modules along existing seams: continue the task_attempts/ submodule pattern for workflows.rs — extract DTOs into a workflows_dto-style file (already exists) and peel handler groups (lifecycle: prepare/start/pause/resume/stop; merge; chat) into workflows/ submodules. ### Data layer (crates/db/src/models/*, crates/db/migrations/*)

The data layer is functional and mostly well-documented, but it carries a visible fault line: two coexisting model lineages with divergent conventions. The original "vibe-kanban" tables (projects, tasks, workspaces, sessions, execution_processes, merges, repos) use BLOB/Uuid primary keys, typed enums (TaskStatus), and CHECK constraints; the newer SoloDawn orchestration tables (workflow, workflow_task, terminal, git_event, concierge, planning_draft, orchestrator_message, quality_*) use TEXT/String primary keys, stringly-typed status columns with NO CHECK constraints, and enums that exist in Rust but are only enforced at the application layer. Cross-lineage foreign keys (workflow_task.vk_task_id BLOB REFERENCES tasks(id), terminal.vk_session_id BLOB REFERENCES sessions(id)) bridge the two id representations. 111 migrations show real discipline in places (indexes, CASCADE retrofits, encryption) but also several full table-rebuild migrations with documented partial-failure risk and one mis-paired up/down file. The good news for extensibility: agents and models are first-class DATA (cli_type + model_config rows), not code, so adding a new coding agent is a data operation, not a schema change. Overall: intelligible and extensible, but the id-type and status-typing split is the main long-term maintainability tax.

**Strengths:** Agents and models are first-class data (cli_type + model_config seed tables with CASCADE and uniqueness), so new agents/models integrate via INSERT with zero schema/code change — directly satisfies swappable-models and rapid-integration goals.; Centralized AES-256-GCM secret handling in encryption.rs (deduplicated from 6 models) with layered key precedence and secrets marked #[serde(skip)]/#[ts(skip)] so keys never reach TS types or the frontend.; Foreign keys and WAL are correctly enabled per-connection (lib.rs:244-252) with explicit CORE-010/CORE-011 rationale, and cascade behavior is verified by real integration tests against in-memory SQLite.; Migrations show genuine operational discipline: performance/composite indexes, CASCADE and SET NULL retrofits, and a startup verify_schema self-heal for stale-compiled-migration drift.; Per-file documentation quality is high — design decisions (e.g. workflow.status as String, dynamic-query convention) are explained inline, aiding 1-month intelligibility.

**Risks:** Two divergent id conventions: BLOB/Uuid PKs (tasks, projects, merges, execution_processes) vs TEXT/String PKs (workflow, terminal, git_event, concierge, quality_*). workflow rows even mix both (id:String, project_id:Uuid), and cross-lineage FKs (workflow_task.vk_task_id BLOB REFERENCES tasks(id), terminal.vk_session_id) straddle the boundary — a permanent source of confusion and impedance for any table spanning both.; Workflow-lineage status columns are unconstrained TEXT with NO CHECK constraint, while old tables (tasks, execution_processes, merges) and even the newest (custom_rules, cli_install_history) DO use CHECK(status IN(...)). update_status(id, status:&str) (workflow.rs:651, terminal.rs:1082) writes arbitrary strings with no validation, so a typo'd status is silently persisted; typed enums (WorkflowStatus etc.) exist but are bypassed at the DB boundary.; Several migrations use the full table-rebuild pattern (CREATE _new / INSERT SELECT / DROP / RENAME) with PRAGMA foreign_keys=OFF inside a sqlx transaction (20260202090000, 20260208020000, 20260417020002). 20260202090000 itself documents that a partial failure could leave the DB with no workflow table, and that the TEXT->BLOB UUID coercion assumes all rows are valid 16-byte UUIDs.; Migration up/down discipline is inconsistent: only 4 pairs use .up.sql/.down.sql while 102 are up-only plain .sql, and 20260316100000_create_system_settings ships a .sql plus a mismatched .down.sql — sqlx will not treat the plain .sql as reversible, so that down file is effectively dead/misleading.; Magic status strings are hardcoded in raw SQL (terminal.rs:646-650 references 'review_passed'/'review_rejected'; git_event process_status defaulted via lower('PENDING')), so the authoritative status vocabulary is scattered across SQL literals, Rust enums, and doc comments rather than defined in one place.

**Recommendations:** Write a one-page ADR in crates/db (e.g. crates/db/DATA_MODEL.md) that names the two lineages, states the rule 'legacy tables use BLOB Uuid PKs, orchestration tables use TEXT String PKs', and lists the bridging FKs (vk_task_id, vk_session_id). This is the highest-leverage, lowest-risk fix — it makes the split intelligible without touching data.; Add CHECK(status IN (...)) constraints to workflow, workflow_task, terminal, and git_event via new table-rebuild migrations (matching the existing custom_rules/execution_processes precedent), and change update_status signatures to take the WorkflowStatus/TerminalStatus enums (using Display at the bind site) instead of &str, so invalid statuses fail at compile or write time.; Introduce a shared const module or enum ::as_str() for the workflow-lineage status vocabularies and replace the hardcoded SQL literals (terminal.rs:646-650, 'review_passed'/'review_rejected') with references to it, so the valid-status set lives in exactly one place.; Adopt .up.sql/.down.sql consistently for all NEW migrations going forward and delete or correct the orphaned 20260316100000_create_system_settings.down.sql so tooling and readers are not misled about reversibility.; For future table-rebuild migrations, add a pre-check (SELECT ... WHERE length(id) <> 16/32) before the rebuild and always ship a .down.sql, per the runbook note already written into 20260202090000, to reduce the partial-failure blast radius.; Add lightweight test builders/Default impls for Workflow and Terminal (the two ~25-30-field structs constructed by hand in lib.rs tests) so tests stay concise and resilient as columns are added.

### React frontend architecture (frontend/src: stores, hooks, components, lib/api, contexts) — state management, d

The SoloDawn frontend is a large (~600 files) but disciplined codebase built on four coexisting state layers: react-query (server state, the dominant and best-organized layer via lib/api.ts's 23 domain-namespaced API objects and per-domain query-key factories), zustand stores (transport/UI state — wsStore, terminalStore, modelStore, UI-preferences), react-context (~20 providers for cross-cutting UI/selection state), and react-db/electric (used narrowly, only for shared_tasks). The container/view split in ui-new is genuinely well-enforced — only ~2 apparent view violations, both false positives (type-only imports). Data flow for real-time updates is coherent: a singleton workflow-scoped wsStore + a generic useJsonPatchWsStream (immer + rfc6902) + thin invalidation hooks (useWorkflowInvalidation) that translate WS events into react-query cache invalidation. The main maintainability risks are not structural rot but concept fragmentation: three overlapping representations of "which agent/model" (BaseCodingAgent enum, modelStore's ApiType, useCliTypes' CliModel), two live-but-parallel model-config pipelines (/api/models via zustand vs /api/cli_types via react-query), a 30-field 'god' WorkspaceContext, dual 'workspace'/'attempt' terminology for one entity, and a 450-line wizardConfigToCreateRequest transform carrying embedded business rules. The logic will remain intelligible at 1 month and extensible at 6 months for anything routed through lib/api + react-query; new-agent integration is only medium-fast because it touches several hardcoded switch statements.

**Strengths:** lib/api.ts is a clean, discoverable HTTP facade: 23 domain-namespaced API objects (projectsApi, tasksApi, workflowsApi...) plus consistent per-domain query-key factories (taskKeys, workflowKeys, attemptKeys...) — the dominant and most maintainable layer.; Data model is anchored by code-generated shared types (ts-rs/schemars from Rust), so the client/server contract is authoritative and drift-resistant; ~90 shared types imported in lib/api.ts alone.; Container/view separation in ui-new is genuinely enforced — only 2 apparent view-with-state cases, both false positives (type-only imports), showing real discipline.; Real-time data flow is coherent and layered: singleton workflow-scoped wsStore for events + generic useJsonPatchWsStream (immer structural sharing + rfc6902 patches, with careful reconnect/backoff and StrictMode-safe teardown) + thin useWorkflowInvalidation that bridges WS events to react-query invalidation.; Workflow wizard is a model of extensible modularity: numbered Step0..Step6 components paired with parallel validators and a WIZARD_STEPS metadata table with mode-aware visibility.

**Risks:** Concept fragmentation for 'agent/model': three overlapping representations (BaseCodingAgent enum, ModelConfig/ApiType, CliModel) with no reconciling module — the single biggest long-term intelligibility tax.; TWO parallel, live model-config pipelines: modelStore.ts (zustand, /api/models/*, hardcoded DEFAULT_MODELS) used by workflow Step3Models vs useCliTypes.ts (react-query, /api/cli_types/*) used by RuntimeSettingsNew. Divergent behavior and duplicate maintenance.; Hardcoded, dated fallback model IDs in modelStore.DEFAULT_MODELS (e.g. claude-3-5-sonnet-20241022, gpt-4-turbo) will silently age and mislead users when the fetch fails.; WorkspaceContext is a 30-field 'god' context aggregating 6+ hooks; every consumer re-renders on any field change and the file is a merge-conflict and blast-radius hotspot.; 'workspace' vs 'attempt' dual terminology for the same entity (useAttempt returns a Workspace) forces a mental alias and invites bugs when the two drift.

**Recommendations:** Introduce a single agent registry module (e.g. lib/agents.ts) keyed by BaseCodingAgent that holds display name, light/dark icon paths, and default models; refactor getAgentName() and AgentIcon() in components/agents/AgentIcon.tsx to read from it. This collapses new-agent integration to one edit site.; Unify the two model-config pipelines: pick react-query + useCliTypes (already the newer /api/cli_types flow) as canonical, port workflow Step3Models off modelStore.ts, and delete or shrink modelStore to genuine local UI state. This removes the largest fragmentation and the stale DEFAULT_MODELS list.; Split WorkspaceContext (contexts/WorkspaceContext.tsx) into 2-3 focused contexts (e.g. WorkspaceSelectionContext, WorkspaceDiffContext, WorkspaceCommentsContext) or expose it via selector hooks so consumers subscribe only to the slice they use, cutting re-render blast radius and merge contention.; Pick one term for the entity and alias the other at the boundary: keep 'workspace' as the UI term, and rename the underlying hook/type to make the identity explicit (e.g. useWorkspace wrapping useAttempt), or add a short doc comment at every attempt<->workspace seam. Do this incrementally, not as a big rename.; Route the ~17 raw-fetch call sites (modelStore, useCliTypes, workflow Step0/4/5, ModelsSettingsNew, RuntimeSettingsNew) through lib/api's makeRequest/handleApiResponse so error handling, auth, and mocking are uniform.; Tighten the WS event abstraction: remove '& Record<string, unknown>' from WorkflowEventHandlers and derive the eventMap from a single typed table so adding an event is one entry with compile-time exhaustiveness; keep the payload interfaces as-is.

### External connector layer (crates/feishu-connector/*, crates/services/src/services/feishu.rs, chat_connector.rs

The git_host subsystem is a genuinely good abstraction: a small `GitHostProvider` trait, an `enum_dispatch` `GitHostService`, URL-based provider detection, and a shared error/type model with clean per-provider CLI wrappers. GitHub and Azure impls are near-symmetric and a third provider could be added in a well-scoped way. The chat side is the opposite: the `ChatConnector` trait is a dead abstraction (only `FeishuConnector` implements it, and nothing ever consumes it as `dyn`/generic), while the actual Feishu ingestion path in feishu.rs bypasses the trait entirely and hardwires `FeishuMessenger`, provider string constants, Chinese-language UX, and Concierge/session logic into one 953-line struct. A second, unrelated chat path exists as an HTTP webhook route (chat_integrations.rs) hardcoded to `telegram` with no connector implementation, so there are two divergent chat integration mechanisms and no unifying seam. The feishu-connector crate itself (SDK adapter, event normalization, reconnect, messenger) is clean, well-documented, and well-tested. Net: git hosts are swappable and extensible; chat platforms are effectively single-provider with abstraction theater on top.

**Strengths:** git_host is a model of a good swappable abstraction: GitHostProvider trait + enum_dispatch GitHostService + URL detection + shared GitHostError/should_retry() + UnifiedPrComment normalization (crates/services/src/services/git_host/{mod.rs,types.rs,detection.rs}); feishu-connector crate is clean, documented, and well-tested: pure normalization/parse functions with unit tests, defensive SDK-payload handling, and a self-contained reconnect policy with capped exponential backoff + jitter (crates/feishu-connector/src/{events.rs,messages.rs,reconnect.rs,sdk.rs}); Robust real-world defensiveness: Feishu redelivery dedup via bounded LRU SeenEvents, poisoned-mutex recovery, gh exit-code-4 auth detection with string fallback, non-UTF-8 CLI output warnings, temp-file PR bodies to avoid shell escaping (feishu.rs:34-55,270-275; github/cli.rs:144-175,205-240); Consumers select git host purely by URL with zero provider branching, proving the abstraction holds at the call sites (pr_monitor.rs:98, task_attempts/pr.rs:316/457/604); Rationale is documented inline where non-obvious (BusMessage::TerminalMessage reuse at feishu.rs:859-865, connected-flag semantics, deprecation suppression in sdk.rs), aiding 1-month intelligibility

**Risks:** ChatConnector is a dead abstraction: only FeishuConnector implements it and nothing consumes it as dyn/generic, so it provides swappability in name only. The real Feishu pipeline (feishu.rs handle_message_inner et al.) hardwires FeishuMessenger and concrete types, so a second chat platform cannot reuse routing/commands (chat_connector.rs:10, feishu.rs:249-345,926); Two divergent, unreconciled chat integration mechanisms: Feishu via WebSocket+SDK inside services, vs an HTTP-webhook route hardcoded to 'telegram' (CHAT_EVENT_ALLOWED_PROVIDER) in server/routes/chat_integrations.rs with NO Telegram connector implementing ChatConnector anywhere. New chat providers have no single seam to plug into (chat_integrations.rs:37,84-91); FeishuService is a 953-line god-object mixing transport lifecycle, dedup, event routing, ~8 slash commands, LLM model-selection UX, Concierge session orchestration, DB access, and the ChatConnector impl — low testability and high change-coupling (feishu.rs entire file); services/feishu.rs is effectively untestable: private async handlers depend on concrete SqlitePool/SharedMessageBus/FeishuMessenger/ConciergeAgent with no trait seams; command routing and dedup logic have no unit coverage; Stringly-typed inbound parsing: FeishuEvent.event is serde_json::Value and parse_message_event indexes with evt["message"]["chat_id"], so schema drift fails only at runtime (feishu-connector/events.rs:34-73)

**Recommendations:** Make ChatConnector earn its place or delete it: either extend it to model inbound events and route FeishuService through it, or remove the inert trait; move command-dispatch/Concierge routing (feishu.rs:249-345) behind a provider-agnostic router that takes &dyn ChatConnector for replies; Reconcile the two chat paths (chat_integrations.rs telegram webhook vs Feishu WS): share one inbound-message model and one binding/command router instead of duplicating ExternalConversationBinding + provider-string plumbing; Extract a shared git_host retry helper to replace the ~7 duplicated backon retry/when/notify blocks in github/mod.rs and azure/mod.rs so the policy lives in one place; Break up the 953-line FeishuService into transport/lifecycle, a table-driven CommandRouter (replacing the if/else-if chain at feishu.rs:302-329), and the Concierge model-selection flow to improve testability; Introduce a typed serde struct for inbound Feishu messages instead of stringly-typed evt["message"]["chat_id"] indexing in feishu-connector/events.rs so schema drift fails at deserialization; Extract Feishu user-facing Chinese strings into a message/i18n module rather than inlining them across feishu.rs
