## Migration-chain impact of removing the v9 `editor` field (per task)

- `EditorConfig`/`EditorType` are DEFINED in `config/editor/mod.rs` and re-exported through `v2 → v3 → v4 → v5 → v6 → v7 → v8 → v9`. Every `vN::Config` has an `editor: EditorConfig` field and every `from_*_config`/`Default` copies it.
- Deleting the editor **backend** (open/launch/remote-url methods + 3 HTTP routes + FE) is self-contained: keep the `EditorConfig`/`EditorType` *types* (and `EditorConfig::new`/`Default`) so the schema and migrations keep compiling and old on-disk configs keep deserializing.
- Deleting the editor **field** from the schema is a chain-wide edit: touch all 8 struct definitions that carry it + their `from_*`/`Default` + drop the re-exports, and add tolerance for the legacy `editor` JSON key (serde will reject unknown fields only if `deny_unknown_fields`; today it does not, so leftover `editor` keys are simply ignored on read). Lowest-risk path: keep the type, delete only the behavior.

## Invisible features

- **External-IDE launcher (G1)** — `EditorConfig::{open_file,open_file_with_hint,spawn_local,remote_url_with_hint,check_availability,with_override}`. Spawns `code`/`cursor`/`windsurf`/`idea`/`zed`/`xed`/`antigravity` or builds a `vscode://`/`cursor://`/`zed://ssh/` remote URL. Wired to 3 POST `/open-editor` routes + a `check-editor-availability` route + many FE components. USED. This is the named deletion target.
- **Remote SSH editor URLs** — `remote_ssh_host`/`remote_ssh_user` on `EditorConfig` build `vscode-remote+ssh` / `zed://ssh` deep links. Sub-feature of G1; only exercised when those fields are set (no first-class UI seen beyond editor settings).
- **profiles.json auto-backup on v6 migration** — `v6.rs` renames `profiles.json` to `profiles_v5_backup_<ts>.json` the first time a v5 config is upgraded. Silent filesystem side effect; fires once per legacy install.
- **`ProfileVariantLabel` (v4)** — explicitly `// DEPRECATED` but still required to deserialize v4/v5 on-disk configs during chained migration.
