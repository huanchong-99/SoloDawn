# R2 — Orchestration Workspace End-to-End Flow

## 1. Overview

This document traces every step from "settings configured" to "code writing starts", including the exact injection point for a mandatory secondary confirmation popup (quality-gate rules review/edit) between `confirmed` and `materialized`.

---

## 2. Full User Journey

```
Settings → enter /workspaces/create → typing first message → planning chat (multi-turn)
        → spec_ready (LLM produces PLANNING_SPEC JSON) → user clicks "Confirm"
        → [audit plan generated by LLM] → status = confirmed
        → user clicks "Materialize" → Workflow created → auto-start
        → code writing begins immediately
```

### 4.1 Status Transitions

```

                     ↘ cancelled
```

```rust
pub const PLANNING_DRAFT_STATUSES: [&str; 5] = [
    "gathering", "spec_ready", "confirmed", "materialized", "cancelled",
];
```

### 4.2 Route Registration

`crates/server/src/routes/mod.rs:166`:
```rust
.nest("/planning-drafts", planning_drafts::planning_draft_routes())
```

All routes under `/api/planning-drafts/`:
- `POST /` — create
- `GET /` — list
- `GET /{id}` — get
- `PUT /{id}/spec` — update spec (forward-only state transitions enforced)
- `POST /{id}/confirm` — **CONFIRM HANDLER**
- `POST /{id}/materialize` — **MATERIALIZE HANDLER**
- `POST /{id}/feishu-sync`
- `POST|DELETE /{id}/audit-doc`
- `GET|POST /{id}/messages`

### 4.3 `send_message` Handler — Auto-transition gathering → spec_ready

`crates/server/src/routes/planning_drafts.rs:392-653`

When the LLM response contains a fenced JSON block with `"productGoal"`, the handler (lines 531-581):
1. Extracts the JSON PLANNING_SPEC block
2. Calls `PlanningDraft::update_spec()` with `requirement_summary` and `technical_spec`
3. Calls `PlanningDraft::update_status("spec_ready")`

This auto-transition is transparent to the user; they see the spec appear and the UI enters `spec_ready` state.

### 4.4 `confirm_draft` Handler — THE FIRST SEAM

`crates/server/src/routes/planning_drafts.rs:286-390`

Guard: requires `status == "spec_ready" || status == "gathering"` (line 298-303).

Steps (all synchronous, blocking the HTTP response):
1. **Lines 306-312**: Determine `AuditMode` from `audit_doc_path` + `retain_builtin`:
   - No doc → `Builtin`
   - Doc + `retain_builtin=true` → `Merged`
   - Doc + `retain_builtin=false` → `Custom`
2. **Lines 315-332**: Read audit doc content from disk at `audit_docs/{draft_id}/{filename}`
3. **Lines 334-352**: Build LLM client (draft config → Claude Code native fallback)
4. **Lines 340-352**: Call `generate_audit_plan()` — LLM call to tailor/merge scoring rubric
5. **Lines 355-370**: Serialize `AuditPlan` to JSON; store via `PlanningDraft::update_audit_plan()`
6. **Line 373**: Call `PlanningDraft::set_confirmed()` → sets status to `"confirmed"`, records `confirmed_at`
7. Returns the updated draft

The `audit_plan` JSON is stored on `planning_draft.audit_plan` column. The `audit_mode` column is updated.

### 4.5 `materialize_draft` Handler — THE SECOND SEAM (code writing starts here)

`crates/server/src/routes/planning_drafts.rs:904-1034`

Guard: requires `status == "confirmed"` (line 915-919).

Steps:
1. **Lines 922-985**: Create a `Workflow` record with:
   - `execution_mode = "agent_planned"`
   - `orchestrator_enabled = true`
   - `initial_goal` = `requirement_summary + technical_spec`
   - `audit_plan = draft.audit_plan.clone()` (line 984) — carries the audit plan forward
2. **Line 996**: `Workflow::create()` — persists workflow to DB
3. **Line 1000**: `PlanningDraft::set_materialized()` — sets draft status to `"materialized"`, stores `materialized_workflow_id`
4. **Lines 1012-1027**: `tokio::spawn` async block calling `auto_prepare_and_start(deployment, &workflow_id)`

### 4.6 `auto_prepare_and_start` — Code Writing Begins

`crates/server/src/routes/workflows.rs:2307-2325`

```rust
pub async fn auto_prepare_and_start(
    deployment: DeploymentImpl,
    workflow_id: &str,
) -> Result<(), AutoStartError> {
    let _ = prepare_workflow(State(deployment.clone()), Path(wf_uuid)).await?;
    tokio::time::sleep(std::time::Duration::from_secs(2)).await;
    let _ = start_workflow(State(deployment), Path(wf_uuid)).await?;
    Ok(())
}
```

`start_workflow` → `OrchestratorRuntime::start_workflow()` → the Orchestrator Agent begins analyzing the spec and creating tasks/terminals → code agents start writing.

**The HTTP response for `/materialize` returns before the auto-start completes** (it's spawned in a background task). The frontend receives `MaterializeResponse { workflow_id }` and navigates to `/board?workflowId=...` immediately.

## 5. Audit Mode / auditPlan / auditDocPath Flow

### Pre-confirm (gathering/spec_ready phase)
- User can upload an audit document via `AuditDocPanel` (rendered in `CreateChatBoxContainer` lines 811-820)
- `POST /api/planning-drafts/{id}/audit-doc` → stores file to disk, saves relative path in `audit_doc_path`
- **BUG**: Frontend (`api.ts:607`) appends multipart field as `audit_doc`; backend (`planning_drafts.rs:1072`) expects field named `file`. Upload is currently broken — the backend will always return "No 'file' field found in multipart upload".
- User sets `retainBuiltin` checkbox in `AuditDocPanel` (line 254)

### At confirm
- `retainBuiltin` is passed from `CreateChatBoxContainer` (line 322-324) → `confirmMutation` → `planningDraftsApi.confirm(draftId, retainBuiltin)` → `POST /confirm` body `{ retainBuiltin }`
- Backend computes `AuditMode`, calls `generate_audit_plan()`, stores JSON result in `planning_draft.audit_plan`
- The audit plan is generated silently — the user never sees it before the status transitions to `confirmed`

### At materialize
- `audit_plan` JSON is copied from `planning_draft` to `workflow.audit_plan` (line 984)
- During code review (`OrchestratorAgent::run_acceptance_review`, `agent.rs:5386-5393`), the orchestrator loads `workflow.audit_plan`, deserializes it, and uses its `raw_principles` to build a scoring review prompt

## 6. The Exact Injection Point for Secondary Confirmation Popup

### Where to add the popup (Frontend)

**File**: `frontend/src/components/ui-new/containers/CreateChatBoxContainer.tsx`

**Location**: The `handleMaterialize` callback (extracted in `usePlanningDraftActions`, line 334-349), which is called when the "Materialize" button is clicked (line 241 in `PlanningStatusBar`).

**Current code** (line 334-349):
```typescript
const handleMaterialize = useCallback(async () => {
  if (!planningDraftId) return;
  try {
    const result = await materializeMutation.mutateAsync(planningDraftId);
    setMaterializedWorkflowId(result.workflowId);
  } catch (e) { ... }
}, [...]);
```

**Injection point**: Before calling `materializeMutation.mutateAsync()`, show a modal that:
1. Displays the `planningDraft.auditPlan` (deserialized JSON) — the 5-dimension scoring rubric
2. Allows editing of scoring criteria, pass threshold, etc.
3. Has a "Proceed with these quality gates" button and "Edit before proceeding" option
4. Only proceeds to `materializeMutation.mutateAsync()` on confirmation

The `planningDraft` object (available at line 426 via `usePlanningDraft(planningDraftId)`) contains `auditPlan: string | null` and `auditMode: 'builtin' | 'merged' | 'custom'`.

**Also needed in the simpler path**: `frontend/src/components/ui-new/containers/PlanningChatContainer.tsx:137` — same `handleMaterialize` pattern.

### Where to add backend gate (optional, for enforcement)

If the popup should be mandatory (not bypassable by calling the API directly), a new field such as `audit_plan_confirmed: bool` could be added to `planning_draft`, set to `true` by a new endpoint `POST /api/planning-drafts/{id}/confirm-audit-plan`, and checked in `materialize_draft` before proceeding.

Current backend guard is only: `draft.status != "confirmed"` (line 915). No check that the user has reviewed the audit plan.

## 7. Between Confirm and Materialize Today

Currently, there is **no UI step** between the two states. After the user clicks "Confirm":
1. Backend generates the audit plan silently (LLM call)
2. Status transitions to `confirmed`
3. React Query refetches the draft
4. `isConfirmed = true` → the "Materialize" button appears immediately in `PlanningStatusBar` (line 239-249)
5. User can click "Materialize" instantly — audit plan is never shown

The only related UI is the `AuditDocPanel` which lets the user upload a custom doc BEFORE confirming, and the `retainBuiltin` toggle. There is no step where the generated audit plan is displayed or editable.

## 8. Settings Configuration Path

Users configure the planner model in Settings → the `workflow_model_library` config array. The `CreateChatBoxContainer` reads this via `extractPlannerModelConfig(config)` (line 135-143) and passes the first available model to `planningDraftsApi.create()`.

The backend auto-fills missing planner config from `workflow_model_library` during `send_message` (lines 417-481 in `planning_drafts.rs`) — priority: anthropic > openai-compatible > any.

## 9. Known Bugs and Risks

1. **Audit doc upload broken** (field name mismatch):
   - Frontend (`api.ts:607`): `formData.append('audit_doc', file)`
   - Backend (`planning_drafts.rs:1072`): `if field_name != "file" { continue; }`
   - The backend never finds the field; returns HTTP 400 "No 'file' field found".

2. **Audit plan not shown to user before materialization**: The `auditPlan` JSON is stored on the draft after confirm, but the frontend never renders it. Users have no visibility into what quality gates will be applied.

3. **`PlanningChatContainer` does not pass `retainBuiltin`** (`PlanningChatContainer.tsx:127-133`): confirms with `{ draftId }` only, so `retainBuiltin` always defaults to `true` on the server.

4. **`usePlanningDraft` stops polling after `confirmed`** (`usePlanningDraft.ts:33-37`): polling stops, so if the audit plan generation is slow, the confirmed state might not refresh automatically. (Mitigated by `queryClient.invalidateQueries` in `useConfirmDraft.onSuccess`.)

5. **`materialize_draft` auto-starts in a background tokio task** without surface to the user. If `auto_prepare_and_start` fails, there's no UI notification — the user navigates to the board but the workflow never starts.
